## 摘要
自托管AI代理通过读写自身的内存和配置文件来运行。代理可能因自身状态被破坏而受到威胁——这种破坏是通过合法的操作系统系统调用来实现的。我们将这类威胁称为自状态攻击。本文研究了操作系统对此类攻击的抵御能力。我们形式化地刻画了一个四维攻击空间（目标、机制、粒度、时序）；研究了预防、检测和恢复的结构性限制；并引入了基于工作负载的可检测性视角。为了实例化该框架，我们从一个典型的自托管代理收集了在不同工作负载配置下运行的实时活动轨迹，并将攻击空间实现为一个23单元格矩阵，包含43个针对真实自状态文件的具体操作，并将它们注入到这些轨迹中。然后我们评估了规范防御策略和基于工作负载的防御策略。实验结果表明，分层防御堆栈（在指令层和配置层上进行访问控制预防，在内存层上进行基于工作负载的检测，以及定期备份用于恢复）对大多数攻击单元有效，但仍有少量残余攻击面在操作系统层面结构上无法区分。这些发现表明，针对新确立的自状态攻击类别，操作系统级防御需要重新思考，这可能为该领域开辟新的研究方向。

## Abstract
Self-hosted AI agents read and write their own memory and configuration files to function. An agent may get compromised via corruption of its own state -- a compromise realized via legitimate OS system call invocation. We refer to this class of threats as self-state attacks. In this paper, we investigate the OS resilience to this class of attacks. Formally, we characterize a four-axis attack space (Target, Mechanism, Granularity, Temporal); investigate the structural limits of prevention, detection, and recovery; and introduce a workload-conditioned view of detectability. To instantiate the framework, we collect live activity traces from a representative self-hosted agent running across distinct workload profiles, and realize the attack space as a 23-cell matrix, 43 concrete operations on real self-state files, and injected into those traces. We then evaluate both canonical and workload-conditioned defense strategies. The empirical results show that a layered defense stack (access-control prevention on the instruction and configuration layers, workload-conditioned detection on the memory layer, and periodic backup for recovery) is effective on most attack cells while a small residual attack surface remains structurally indistinguishable at the OS level. These findings suggest that against the newly established class of self-state attacks, OS-level defense needs to be reconsidered, potentially opening new research directions in the field.

### 2. 论文提出的方法论
- **威胁模型形式化**：将自状态攻击定义为攻击操作集合 A(R) 是同一资源 R 上合法操作集合 L(R) 的子集（即 A(R)⊆L(R)），因此在 VFS 边界上攻击与合法写操作使用相同的系统调用元组（目标路径、操作类型、大小增量 δs、模式变化、时间间隔 Δt）。
- **四维攻击空间**：从 OS 可观测特征出发构建攻击分类法：
  - **目标（Target）**：指令层（Instruction）、记忆层（Memory）、配置层（Config）。
  - **机制（Mechanism）**：修改（Modify）、添加（Add）、删除（Delete）、拒绝访问（Deny）。
  - **粒度（Granularity）**：整文件（G1）、大增量（G2）、行级（G3）、最小增量（≤4 字节，G4）。
  - **时序（Temporal）**：单次（T1）、突发对齐（T2）、静默间隙（T3）、慢滴（T4）。  
  由此生成 **23 个规范攻击单元**（Target×Mechanism×Granularity）并将其细化为 43 个具体攻击脚本。
- **OS 防御的结构性限制分析**：
  - **预防（Prevention）**：由于 A(R) 和 L(R) 在操作层次不可区分，任何基于权限的策略若阻止攻击也会阻止合法自我更新，无法仅凭事件级策略完全遏制。
  - **检测（Detection）**：OS 级监控（如 inotify、auditd）只能观察到元组（操作类型、大小增量、间隔），无法判断写入内容的语义有害性，因此在特征空间上恶意与良性写入可能重叠，形成不可区分区域。
  - **恢复（Recovery）**：依赖于检测的准确性，且存在数据丢失与污染回溯的模糊性。
- **工作负载条件检测（Workload‑conditioned detection）**：提出根据代理不同工作负载配置，为每个文件（类型）学习特定操作基线（μs, σs, μt, σt），计算每个事件的 z‑score（大小增量和 log 时间间隔的标准化偏差的最大值），当 score 超过阈值 τ=2.0 时触发告警。依据不同工作负载下攻击的检测真阳性率（TPR），将每个攻击单元划入三类：
  - **V（可见，Visible）**：所有配置下 TPR≥0.80；
  - **C（条件，Conditioned）**：最小 TPR 在 [0.50, 0.80) 区间；
  - **I（不可区分，Indistinguishable）**：至少一个配置下 TPR<0.50。

### 3. 实验设计
- **实验对象与工作负载**：以自托管代理 **OpenClaw** 为核心构建轻量级脚手架 `openclaw‑core`，模拟真实代理的自我状态 I/O。设计四种工作负载插槽：
  - **W1 编码助手**（来自 Aider 多语种编程任务）；
  - **W2 研究助手**（来自 FRAMES 知识推理任务）；
  - **W3 运维代理**（作者构建的自我配置任务）；
  - **W4 通用助手**（作者构建的日常辅助任务）。
- **轨迹收集**：使用 Linux inotify 记录工作负载过程中代理产生的所有 VFS 事件（CREATE, MODIFY, DELETE, MOVE, ATTRIB），形成合法的活动日志。每个工作负载运行 5 个不同任务顺序的链条（共 600 个任务槽）以减小顺序偏差。
- **攻击注入**：将 23 个单元的 43 个确定性攻击操作（如修改文件特定字节、删除文件、chmod 000）以合成事件的形式注入到合法轨迹中，模拟攻击事件发生。
- **对比方法**：
  - **预防**：6 级 Unix 权限锁定（L0–L5），逐步禁止指令、配置和记忆层的写入。
  - **检测**：
    - **B0**：基于文件哈希的完整性监控（Wazuh 风格）。
    - **B1**：工作负载盲的统计异常检测（跨所有配置拟合单一基线）。
    - **B2**：工作负载条件统计异常检测（每工作负载分别拟合基线）。
  - **恢复**：基于事件计数的周期性备份恢复，间隔 k∈{5,15,30,60} 事件。

- **充分性与公平性**：实验覆盖了多样的工作负载、攻击粒度和检测策略，并进行了必要的消融（工作负载直、时序感知）与对比（FIM）。使用可重现的轨迹、固定种子和独立试验，多次随机划分数据，降低了数据划分偏差。整体实验设计客观，能充分体现 OS 防御在不同攻击单元下的局限性。不过，受限于个人构建的工作负载（W3、W4）可能无法代表所有真实使用场景，且检测器仅依赖事件元数据，公平性主要体现为统一阈值下的比较。

### 6. 论文的主要结论与发现
- **预防层面**：锁定指令和配置层（L3 策略）可在保持大部分功能的前提下阻止约 61% 的攻击单元；但记忆层的写操作是功能必需的，无法单纯通过权限控制彻底防御。
- **检测层面**：工作负载条件检测（B2）在 τ=2.0 时将 23 个攻击单元分为：
  - **11 个 V 单元**（例如整文件替换、删除，或指令/配置文件的写）；
  - **8 个 C 单元**（如配置文件小修改、特定用户文件编辑，在有的工作负载下可检测）；
  - **4 个 I 单元**（记忆层的小修改或每日日志追加，在运维类工作负载下与合法写入重叠，不可区分）。
- **时序影响**：攻击时机（突发、静默）对大部分单元格的检测等级影响微小，慢滴式写入（T4）因单次事件均落在基线内而无法按事件检测。
- **恢复能力**：备份恢复可以在检测触发后完全修复所有非自毁性的攻击，但数据丢失量与备份间隔成正比；若攻击同时破坏备份存储，恢复将完全失败。
- **分层防御效果**：采用 L3 预防 + B2 检测 + 15 事件间隔备份的堆栈，可将 23 个攻击单元中除 4 个 I 单元外的所有攻击有效闭环，剩余的攻击面在 OS 层结构上无法区分，需依赖更上层的语义防御。

### 7. 优点
- **问题定义清晰**：首次将“自状态攻击”提炼为一类有明确定义和形式化模型的安全问题，连接了提示注入和 OS 安全两个领域。
- **系统化的攻击空间分析**：四维攻击分类法不仅覆盖了现有威胁目录（MITRE ATLAS、ATT&CK、OWASP）中的模式，还通过粒度和时序轴揭示了防御可观测性的细微差异。
- **结构限制的严谨论证**：从 OS 防御的可观测特征出发，推导出预防、检测、恢复各自的理论瓶颈，并通过实验实证了这些瓶颈的存在。
- **工作负载条件检测的提出**：通过在具体工作负载上建立文件级行为基线，使部分原本被全局基线掩盖的攻击得以检测，展现了适应不同部署环境的可能性。
- **实验设计扎实**：基于真实代理的轨迹、多种工作负载、大量试验次数和消融对比，结果具有说服力。开源了完整的实验平台和攻击矩阵，增强了可复现性。

### 8. 不足与局限
- **工作负载的全面性**：仅使用了四种手动划分或从特定基准改编的工作负载，可能无法覆盖所有自托管代理的真实使用模式（如多代理协作、高频率心跳任务等）。某些攻击单元的 V/C 类别可能随工作负载数量增加而迁移。
- **检测方法较为简单**：异常检测仅基于大小增量和时间间隔的 z‑score，忽略文件内容、系统调用序列等更丰富的特征，可能导致部分 C 单元在更复杂的模型中变为 I 单元，或反之。
- **单代理依赖性**：实验仅基于 OpenClaw 的简化和受控环境，虽然声称其自状态契约具有代表性，但其他代理（如 Claude Code）的实现细节（如文件更新频率、工具调用模式）可能影响结果的外推性。
- **“慢滴”攻击处理不足**：论文指出慢滴式攻击因每个事件均不触警而无法被单事件检测识别，但未深入探讨多事件累积检测或时序建模的可能性。
- **语义层与 OS 层的衔接**：研究明确将攻击限定在 OS 可观测层，未考虑与语义层防御（如 GuardAgent、输出过滤）的互补关系或组合后的整体安全效果。
- **真实攻击载荷的有限性**：攻击脚本虽具有代表性，但均为确定性文件操作，未考虑攻击者可能使用更隐蔽的语义转换（如同义词替换）来进一步降低异常分数，从而可能扩大 I 单元的范围。
