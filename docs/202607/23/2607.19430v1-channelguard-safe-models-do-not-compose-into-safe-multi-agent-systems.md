## 摘要
多智能体大语言模型应用将规划器、工作智能体、验证器和合成器链接起来，智能体之间的每一次跳转都是一个未受监控的通道，对手可以通过这些通道偷运指令。现有的防御措施仅保护输入边界（如IBProtector、Llama Guard、困惑度过滤器、SmoothLLM），或在应用外部作为不透明、随机的提供商端过滤器运行。我们展示了这一差距带来了很少被衡量的后果：在涵盖8类攻击、5种防御和3个模型后端的2100条跟踪评估中，一个在标准报告下显得完全安全的无防御流水线（针对工具和内存中毒的攻击成功率为0.000）几乎完全归功于云提供商的服务器端过滤器（在Azure GPT-5上60次攻击中有54次被拦截），而在没有此类过滤器的后端上，则悄悄转向依赖智能体模型自身的对齐。仅报告结果掩盖了这种依赖性。我们提出了ChannelGuard，一个无需训练的纵深防御框架，它在每个智能体间通道上放置信息瓶颈门；每个门通过嵌入相似度比对通道文本与对抗短语库进行评分，并确定性地放行、压缩或拦截，无需额外的大语言模型调用，同时通过归因方法记录每一层阻止了哪次攻击。ChannelGuard的工具输出门在应用层拦截了30次工具中毒攻击中的全部30次，在Azure GPT-5、Anthropic Sonnet 4.5和Anthropic Haiku 4.5上表现一致，而无防御流水线则在各后端之间完全不同；它还将提示注入攻击成功率降低了一半（从0.333降至0.167），并完全保持了GSM8K的准确率（0.867）。白盒自适应改写可以规避每个嵌入门，而扰动投票基线表现更好。扩展附录增加了基线、消融实验、参数扫描、良性保留分析以及一名评判审计（kappa = 0.900），总成本为47.36美元。

## Abstract
Multi-agent LLM applications chain a planner, worker agents, a verifier, and a synthesizer, and every hop between agents is an unmonitored channel through which an adversary can smuggle instructions. Existing defenses guard only the input boundary (IBProtector, Llama Guard, perplexity filters, SmoothLLM) or run outside the application as opaque, stochastic provider-side filters. We show this gap carries a consequence rarely measured: on a 2,100-trace evaluation across eight attack families, five defenses, and three model backends, an undefended pipeline that appears fully safe under standard reporting (attack success 0.000 on tool- and memory-poisoning) owes that safety almost entirely to the cloud provider's server-side filter (54 of 60 blocks on Azure GPT-5), and silently shifts to the agent model's own alignment on a backend without such a filter. Outcome-only reporting hides this dependence. We present ChannelGuard, a training-free defense-in-depth framework placing information-bottleneck gates on every inter-agent channel; each scores channel text against an adversarial phrase bank by embedding similarity and deterministically passes, compresses, or blocks it, adding no LLM call, while an attribution method records which layer stopped each attack. ChannelGuard's tool-output gate blocks Tool Poisoning 30 of 30 at the application layer, identically across Azure GPT-5, Anthropic Sonnet 4.5, and Anthropic Haiku 4.5, whereas the undefended pipeline shifts entirely across backends; it also lowers Prompt Injection attack success by half (0.333 to 0.167) and preserves GSM8K accuracy exactly (0.867). White-box adaptive paraphrase evades every embedding gate, where a perturb-and-vote baseline does better. An extended appendix adds baselines, ablations, sweeps, a benign-preservation analysis, and a judge audit (kappa = 0.900), at a total cost of 47.36 USD.

### 1. 论文的核心问题与整体背景
- **核心问题**：多智能体大语言模型（LLM）系统将规划器、工作者、验证器、合成器等组件串联，但智能体之间的每一次文本传递都是一个**未被监控的通道**。攻击者可在任意通道（如工具返回、共享内存、用户输入）中注入恶意指令，并使其向下游传播，最终污染系统输出。
- **差距与隐患**：现有防御几乎全部集中在**输入边界**（如用户查询），对内部通道毫无察觉；而云提供商的服务端过滤器虽可拦截部分攻击，却**不透明、随机、不受开发者控制**，且其安全机制会随提供商变化而**悄然转移**（例如从 Azure 过滤器转到模型自身对齐）。仅报告攻击成功率（ASR）的评估标准掩盖了这一依赖关系。
- **整体含义**：安全的单模型组合后并不自动构成安全的多智能体系统，业界亟需一种**应用层可控、通道级纵深防御**，并能**测量和归因**安全机制的来源。

### 2. 方法论：ChannelGuard 框架
- **核心思路**：在流水线中**六个关键通道**处各插入一个信息瓶颈（IB）门，对通过文本进行风险评分，并根据阈值决定**放行、压缩或阻塞**，不额外调用大语言模型，同时记录每次决策以实现攻击阻断归因。
- **关键技术细节**：
  - **门机制**（复用相同算法，放置于不同位置）：
    1. 将通道文本按句号等终端标点拆分为句子。
    2. 用本地 CPU 运行的 `all-MiniLM-L6-v2` 对每个句子与预定义的**对抗短语库**（20条）计算最大余弦相似度，得到该文本的风险分数 `risk(t)`。
    3. 设定两个阈值：阻塞阈值 \(\theta\)（默认 0.60）和压缩阈值 \(\alpha \theta\)（默认 \(\alpha=0.70\)）。  
       - 若 \(\text{risk} \ge \theta\) → **阻塞**（返回空字符串）；  
       - 若 \(\alpha\theta \le \text{risk} < \theta\) → **压缩**（只保留前 \(N=2\) 句）；  
       - 若 \(\text{risk} < \alpha\theta\) → **放行**（原文不变）。
  - **六大门的位置**（覆盖所有内部攻击面）：
    | 门编号 | 保护通道 | 是否输入边界？ |
    |--------|----------|----------------|
    | IB‑0   | 用户→规划器（输入） | 是（仅此一个） |
    | IB‑1   | 规划器→每名工作者（子任务拆分） | 否 |
    | IB‑2/IB‑4 | 工作者输出→验证器（预验证） | 否 |
    | IB‑3   | 工具→工作者（工具输出） | 否 |
    | IB‑3‑mem | 共享内存→工作者（内存读取） | 否 |
    | IB‑5   | 全体工作者输出→合成器 | 否 |
  - **验证器与合成器**：被压缩或放行的工作者回答经**不确定性感知验证器**（5 次蒙特卡洛重采样，Claude Sonnet 4.5）评分；若平均安全分过低且方差大则拒绝；合成器前再经一次门过滤，若无存留回答则硬编码拒绝。
  - **可调性与理论基础**：引入严格度参数 \(\beta\) 可统一缩放有效阈值；门的形式化性质包括三元输出信息边界 ≤ log₂3 bit、嵌入空间扰动一阶裕度等。
  - **攻击归因**：每条 trace 记录“最先拦截层”（Leke/IB/Provider Filter/Verifier/Synth Refusal/Safe），用于揭示安全机制的真正来源，而非仅看结果 ASR。

### 6. 主要结论与发现
- **工具中毒绝对保护**：ChannelGuard 的 IB‑3 门跨三个后端（Azure GPT‑5，Sonnet，Haiku）均完全阻断 30/30 工具中毒，而无防御系统在 Azure 上依赖服务端过滤，在无此过滤器的后端则完全依赖模型自身对齐——**安全机制从提供者侧“取代”为应用侧**。
- **提示注入风险减半**：主 Azure 堆栈上，ASR 从 0.333 降至 0.167（相对降 50%），但统计上未达强显著（McNemar \(p=0.125\)，n=30），方向性明确。
- **归因揭示隐藏依赖**：无防御系统看似 ASR=0 的攻击（如工具/内存中毒），实际 90% 的拦截来自 Azure 过滤器，而非应用自身能力。这种**机制替代**在常规 ASR 报告中完全不可见。
- **推理能力无损**：GSM8K 准确率两者均为 0.867，即门不对良性任务引入干扰。
- **效率提升**：在早期门短路拦截的攻击（如提示注入）上，ChannelGuard 比无防御快 3.3 倍（因为直接阻塞而不调用下游大模型），整体攻击混合负载下快 1.19 倍。
- **明确的失败边界**：  
  - 静态短语库无法防御**白盒自适应改述**（ASR 仍高达 0.667）；  
  - **叙事型内存攻击**（如慢速漂移）因相似度不足而无法触发门，安全全仰仗模型自身。

### 7. 优点
- **填补关键空白**：首次全面覆盖多智能体间通道的防御，将保护面从输入扩展至工具、内存、子任务等非输入交互点。
- **简易与低成本**：无需训练、无额外 LLM 调用，轻量嵌入+短语匹配即插即用，成本低廉并可完全离线运行，不影响原有大模型。
- **透明且可归因**：每条 trace 记录拦截层，赋予开发者对“谁在保护系统”的审计能力，利于可靠性与合规审计。
- **提供者无关**：关键通道（如工具输出）的决策仅依赖通道文本和短语库，不依赖后端模型，安全性稳定可预期。
- **实验严谨性高**：配对设计、大量基线、良性保留测试、随机化检验、交叉评判审计、完整的代码和 trace 开源，可复现性强。

### 8. 不足与局限
- **对自适应攻击脆弱**：白盒知晓短语库的改写可轻松绕过所有嵌入门，SmoothLLM 等非签名式防御表现更佳（ASR 仅 0.200），静态签名防御的固有天花板显现。
- **叙事注入盲区**：攻击载荷伪装在正常对话中（如“慢速漂移”），短语相似度低导致门无法触发，安全性完全依赖模型自身行为，未能提供额外保障。
- **压缩带泄漏**：在用户输入处的 IB‑0 门，压缩仅保留前两句话的策略导致约 23% 的压缩样本仍泄漏，因为用户可能将恶意指令放在前部。
- **样本量局限**：主 ASR 对比基于 30 样本的单一种子，部分统计结论（如提示注入的显著性）尚不牢固。
- **短语库泛化能力不足**：手工编写的静态短语库覆盖面有限，攻击越复杂，短语匹配越可能失效。
- **实验设定简化**：使用模拟工具（canned responses），尚未在真实检索增强（RAG）等动态工具上验证；评估语言仅为英文。
- **验证器盲点**：当前过滤规则忽略“安全但高不确定性”的样本，该类样本中有 19.5% 的 trace 最终泄漏，存在遗漏风险。
