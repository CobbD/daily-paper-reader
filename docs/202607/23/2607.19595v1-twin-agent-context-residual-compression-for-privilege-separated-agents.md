## 摘要
大型语言模型（LLM）代理容易受到安全风险的影响，例如来自不可信上下文的提示注入攻击，这些攻击会操纵下游推理和工具使用。现有的安全设计方法通过将不可信观察与特权执行分离并谨慎控制信息流来缓解这一风险，但通常会降低实用性，并需要大量的任务特定工程。因此，我们提出了 Twin Agent，这是一种受代理上下文中残差编码启发的通用权限分离设计模式。Twin Agent 由两个几乎对称的代理组成：一个负责检查不可信信息的探索代理（Explore Agent），以及一个负责执行特权操作的安全代理（Safe Agent）。探索代理以安全代理的当前上下文为条件，并仅向安全代理传递关于下一步操作的紧凑提示。这种设计减少了保持任务实用性所需的信息量，从而实现了更好的安全-实用性权衡，我们通过测量随着提示长度的变化，实用性和攻击成功率如何变化，对此进行了实证验证。我们在使用 SWE-bench Lite 的长期软件工程任务上，以及使用 AgentDojo 和 DecodingTrust-Agent 的异构多工具交互任务上评估了 Twin Agent。在这两个基准测试中，Twin Agent 在防止提示注入攻击的同时保持了高任务实用性，其表现优于未设防的代理和权限分离基线方法。

## Abstract
Large language model (LLM) agents are vulnerable to security risks, such as prompt injection attacks from untrusted context that manipulate downstream reasoning and tool use. Existing secure-by-design approaches mitigate this risk by separating untrusted observations from privileged execution and careful control of information flow, but often degrade utility and require extensive task-specific engineering. We thus propose Twin Agent, a general privilege separation design pattern inspired by residual coding in the agent context. Twin Agent consists of two nearly symmetric agents: an Explore Agent that inspects untrusted information and a Safe Agent that executes privileged actions. The Explore Agent is conditioned on the Safe Agent's current context and communicates only compact hints to the Safe Agent about the next action to take. This design reduces the information needed to preserve task utility and thus achieves a better security--utility tradeoff, which we empirically verify by measuring how utility and attack success change as the length of hints varies. We evaluate Twin Agent on long-horizon software engineering tasks with SWE-bench Lite and on heterogeneous multi-tool interaction tasks with AgentDojo and DecodingTrust-Agent. Across both benchmarks, Twin Agent preserves high task utility while preventing prompt injection attacks, outperforming both undefended agents and privilege separation baselines.

# Twin Agent：面向权限分离代理的上下文残差压缩 详细总结

## 1. 核心问题与整体含义
- **研究背景**：大型语言模型（LLM）代理在真实环境中需处理不可信信息（如网页、工具输出、用户提交的 Issue），同时执行特权操作（代码执行、文件系统访问等）。
- **核心问题**：不可信输入可通过提示注入攻击操控代理的推理与工具使用，导致有害动作（如删除文件、写入后门）。现有安全设计（如严格权限分离、计划–执行模式）虽能降低攻击成功率，但常常严重损害任务完成率（utility），并需要大量任务特化的工程改造。
- **整体含义**：论文提出 **Twin Agent** 架构，将安全代理的设计看作一个**压缩信息流控制问题**——尽可能减少从不可信侧流向特权侧的信息量，保留足够的任务相关信息即可，从而在不牺牲效用的情况下防御注入攻击。

## 2. 方法论
- **核心思想**：受残差编码（residual coding）启发，将特权分离中的通信变为“差分”而非“绝对”。
  - **探索代理**（Explore Agent）：获取不可信区域 \( U \) 的全部内容，但其动作仅限于观察，**不能执行特权操作**。
  - **安全代理**（Safe Agent）：执行特权操作（如 bash 命令），但**不能直接观察不可信数据**，只接收探索代理传来的紧凑提示（hint）。
- **工作机制**：
  - 在每个时间步，探索代理观察安全代理当前的交互历史 \( I^{(s)}_t \)，并基于可观察的全部上下文（含不可信信息）生成提示 \( H^{(e)}_t \)。
  - 安全代理依据提示（以及可信信息）做出下一步决策，**不接触原始不可信内容**。
  - 提示长度受预算 \( b \) 限制，从而控制跨信任边界的信息流量。
- **设计目标**：最小化提示长度的同时，使得安全代理的动作尽可能接近无防御原始代理的动作（式（11））。
- **安全机制**：
  - 限制提示长度（如 SWE‑bench 上限制 100 字符），使其难以携带完整的注入指令。
  - 在安全代理侧添加**提示注入检测器**（一个简单的分类器）和**提醒文本**，拦截或警告可疑提示。
  - 整体假设：前沿模型很难被极短的注入攻击成功操控，且这些短注入必须同时绕过检测器。
- **通用性**：该框架是现有 ReAct 代理的“即插即用”替代，仅需轻微修改提示结构，无需重新设计工作流。

## 3. 实验设计
- **评估基准与场景**：
  1. **SWE‑bench‑injected**：从 SWE‑bench Lite 中筛选 232 个软件工程调试实例，将恶意指令注入到 Issue 描述中（删除文件或写入秘密字符串），模拟不可信用户提交。
  2. **AgentDojo**：包含 workspace、slack、travel、banking 四种多工具交互环境，使用默认的 `important_instructions` 攻击（于工具输出中注入恶意指令）。
  3. **DecodingTrust‑Agent**：选择 CRM 和 Finance 核心套件，共 60 个任务实例，并在工具响应中内嵌提示注入攻击。
- **对比方法**：
  - **Undefended Agent**：无任何防御的原始代理。
  - **Dual Agent**：权限分离但不允许被隔离代理向特权代理传递任何信息。
  - **CaMeL**：计划优先的权限分离架构（Debenedetti et al., 2025）。
- **评估指标**：
  - **Utility**：按各基准原有评估协议计算的任务完成率。
  - **Attack Success Rate (ASR)**：注入指令被成功执行的比例。
- **自适应攻击测试**：
  - 针对 SWE‑bench 设计两阶段攻击：阶段一用 PAIR 算法（100 次迭代）优化能绕过检测并诱使安全代理执行恶意动作的提示；阶段二在 Issue 中添加“复制粘贴”注入，强迫探索代理原样传递恶意提示。测试单轮和多轮场景。

## 6. 主要结论与发现
- Twin Agent 在所有基准上**几乎完全阻止了提示注入攻击**（ASR 大多为 0.0%），同时**保持甚至略高于无防御代理的任务效用**（例如 SWE‑bench 上从 61.2% 升至 62.5%）。
- 与严格隔离的 Dual Agent（效用几乎为 0）和计划优先的 CaMeL（效用仅 30.9%）相比，Twin Agent 在安全与实用之间取得了显著的更好平衡。
- 提示长度预算是控制安全–效用折衷的有效旋钮：较小的预算降低攻击面但可能损害效用，较大的预算增加攻击面但提升效用（图 3）。
- 自适应攻击下，安全代理在单轮完全抵御；多轮下 ASR 为 4.7%，但通过多次运行投票可将成功概率降为 0，表明安全性具有一定的随机性但可通过工程手段进一步提升。

## 7. 优点
- **设计简约且通用**：无需为每个任务定制安全工作流，可作为现有代理（如 SWE‑Agent）的直接替换。
- **可控的安全–效用权衡**：通过约束提示长度，系统设计者可直观地在实用性和安全性之间调节。
- **实证全面**：在长期软件工程和异构多工具交互两类任务上均表现出色，超越了现有防御架构。
- **残差压缩思路新颖**：将安全通信问题归纳为压缩信息流问题，为未来研究提供了理论视角。
- **成本可接受**：性能提升的同时仅带来约两倍的成本增加，适合实际部署。

## 8. 不足与局限
- **无形式化安全保障**：安全性依赖于“短提示难以实施成功攻击”的经验假设，未来更强的攻击可能突破。
- **攻击覆盖有限**：仅测试了两种简单的恶意动作（写文件、删文件），未涉及更复杂的多步攻击或逻辑炸弹。
- **自适应攻击范围受限**：PAIR 攻击仅针对单轮优化，多轮攻击的成功可能未完全穷尽；探索代理完全被控（复制粘贴）是最坏假设，但实际场景中攻击者可能更难操控探索代理。
- **信息预算硬限制**：极短的预算可能导致复杂任务信息丢失，效用下降；预算的确定仍需人工调节。
- **评估领域尚窄**：仅在软件工程和少数工具使用环境中测试，在更多开放领域（如电商、医疗）的适用性有待验证。
- **未讨论隐私和侧信道风险**：虽然压缩了信息，提示中仍可能包含敏感内容，未做差分隐私等保护。
