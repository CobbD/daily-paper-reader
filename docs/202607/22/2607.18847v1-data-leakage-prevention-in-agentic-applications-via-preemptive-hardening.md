## 摘要
智能体系统将大语言模型（LLM）驱动的规划与外部工具接口相结合，使得指令/数据边界失效和提示注入攻击可能导致数据泄露和工具误用。在跨多个代码库和异构智能体的工作流中，一致地施加所需控制尤为困难。为解决多智能体系统中的这一挑战，我们提出了一种预部署管道，用于扫描、加固和验证智能体应用。该管道分析提示模板、工具接口及工具调用代码，识别导致泄露的模式并生成可执行的补丁。随后，通过对抗性提示注入攻击和良性输入变体对加固后的应用进行验证，确保缓解措施不会破坏预期行为。在加固阶段，优先处理高风险工具，并应用最小侵入性的缓解措施，包括模式收紧、边界净化、基于允许列表的工具门控以及最小权限检查。在验证阶段，管道自动生成模拟越狱、指令覆盖和针对工具的操控的攻击输入，以及良性任务变体，以确认加固后的应用在修复后仍保留其功能。我们在五个真实世界的智能体应用以及 AgentDojo 基准上评估了该管道。在所有应用中，所提出的管道识别出重复出现的导致泄露的模式，并生成了可在不破坏预期应用行为的情况下集成的补丁。结果表明，在面对基本越狱和指令覆盖攻击时，应用代码的修改消除了泄露，实现了 100% 的泄露减少；在压力诱导的操控条件下，泄露减少了 91%，且无需持续的运行时策略执行。

## Abstract
Agentic systems integrate LLM driven planning with interfaces to external tools, making data leakage and tool misuse feasible via instruction/data boundary failures and prompt injection attacks. Enforcing required controls consistently is particularly challenging in workflows spanning many codebases and heterogeneous agents. To address this challenge in multi agentic systems, we present a pre-deployment pipeline for scanning, hardening, and validation of agentic applications. The pipeline analyzes prompt templates, tool interfaces, and tool-invocation code to identify leakage-enabling patterns and generate actionable patches. The hardened application is then validated through adversarial prompt injection attacks and benign input variations ensuring that mitigations do not disrupt intended behavior. In the hardening stage, high-risk tools are prioritized, and minimally invasive mitigations are applied, including schema tightening, boundary sanitization, allowlist-based tool gating, and least-privilege checks. In the validation stage, the pipeline automatically generates attack inputs that mimic jailbreaks, instruction overrides, and tool-targeted manipulation, along with benign task variants, to confirm that the functionality of the hardened application is preserved after remediation. We evaluated the pipeline on five real-world agentic applications, as well as on the AgentDojo benchmark. Across all applications, the proposed pipeline identified recurring leakage-enabling patterns and generated patches that can be integrated without disrupting the intended application behavior. The resulting modifications of application code were shown to eliminate leaks when targeted by basic jailbreak and instruction-override attacks, achieving a 100% reduction in leakage, and reduce leaks by 91% under conditions of stress-induced manipulation, without the need of continuous runtime policy enforcement.

### 2. 方法论
管道由三大模块顺序执行，并可迭代多轮：

- **分析模块（Analyzer）**
  - **代理发现**：使用 Python AST 解析源码，提取代理、工具接口、记忆访问及代理间通信拓扑。
  - **工具映射**：对每个工具的参数面进行敏感度分级（高/中/低），标记外部汇（exfiltration sink）。
  - **工具汇检查**：结合结构分析和 LLM 语义推理，对数据流节点分配多维敏感标签（如 PII、凭证、内部指令），并基于可达性传播标签，枚举所有泄漏路径。
  - **风险排序**：依据操作危险度、用户可控暴露度、数据敏感度综合打分，输出分层审计报告（高/中/低优先级）。

- **修改模块（Modifier）**
  - 针对审计报告中的高风险路径，应用最小化、可框架兼容的补丁：
    - **重写提示**：注入显式控制头，将不可信内容严格作为数据分隔，禁止作为指令执行。
    - **剥离不安全上下文**：在数据跨越敏感边界前，利用 LLM 仅保留任务必需信息，丢弃凭证、ID 等标签数据。
    - **添加角色分离**：在角色间使用类型化交接，明确“指令”与“数据”字段，阻断跨代理指令注入。
    - **添加能力边界**：按最小权限收紧工具允许列表，强化参数模式（拒绝携带敏感标签的参数），插入运行时守卫。
    - **转换不安全外发请求**：在构造外部调用时，移除敏感字段、归一化目的地，确保外发内容最小化。

- **验证模块（Validation）**
  - 在加固后的应用上同时运行良性任务与对抗性攻击，提供安全‑效用双重检验，并反馈至修改模块进行迭代改进。
  - **对抗分支**：自适应生成四类攻击（直接注入、越狱、指令覆盖、压力诱导），通过应用适配器监控工具调用、外发请求等汇点，用应用专用泄漏检测器判断泄露，维护跨轮次状态以复现历史攻击并验证修复。
  - **良性分支**：验证任务完成率，防止过度加固破坏正常功能。

### 3. 实验设计
实验围绕三个问题：安全性提升、泛化能力、效用保持。

- **测试平台**：五个用 CrewAI / LangGraph 实现的多智能体应用（网络监控助手、HR助手、自动行程规划器、自动邮件回复器、候选人招聘），包含层次/顺序工作流、外部工具、敏感数据等。

- **攻击与指标**：对每个应用比较加固前（B）和加固后（A），攻击分为直接请求、基本越狱、指令覆盖、压力诱导四类。指标为攻击成功率（ASR）和良性任务成功率（BTSR）。

- **优于纯运行时防御**：与 FIDES 比较，本方法在维持或提高安全性的同时，良性任务成功率更高、幻觉更少，凸显了构建时加固在效用保持上的优势。

### 7. 优点
- **前移防护理念**：直接从源码级别消除泄露路径，无需依赖运行时策略强制执行或模型微调，避免了额外推理开销。
- **自动化与框架无关**：管道支持 CrewAI 和 LangGraph，可通过适配器扩展至其他框架，且提供 CI/CD 友好型输出。
- **迭代与闭环**：分析—修改—验证形成闭环，验证模块的自适应攻击生成和效用检测使迭代改进具备实际可操作性。
- **最小侵入**：仅针对高风险点局部修补，不影响无关功能，最大化保留原有任务表现。
- **实验评估扎实**：多应用、多攻击类别、消融与外部对照相结合，较全面地衡量了安全性、泛化能力和效用保持。

### 8. 不足与局限
- **残余泄露**：压力诱导类攻击仍未能完全阻断，在最复杂的网络监控场景下仍有 7.4% 的攻击成功率。
- **分析盲区**：对于保持模式正确但恶意篡改的跨代理工件可能无法检测；对隐蔽性强、基于语义的间接泄露，LLM 推理可能不能全覆盖。
- **多用户安全欠缺**：当前设计对多租户授权、上下文隔离支持不足。
- **自身依赖 LLM**：管道的关键分析、重写步骤均依赖 LLM，其输出质量波动可能影响加固一致性和可靠性，但论文未对此做鲁棒性评估。
- **评估范围有限**：AgentDojo 仅用单一攻击，对比 FIDES 的应用数量较少，未能展示在更多样化攻击模型或更大规模系统下的表现。
- **成本未量化**：缺少对管道运行时间、计算资源消耗的统计分析，难以判断在生产环境中的效率。
