## 摘要
设备端大语言模型推理面临响应延迟、有限硬件资源和用户隐私的三难困境。完全云端推理提供强大算力，但会暴露用户提示和对话数据，而独立的设备端推理对于大多数消费级和嵌入式边缘设备而言并不可行。本文提出了一种以隐私为中心、基于端点认证KV缓存的边缘云协同大语言模型推理框架。本地端点负责输入预处理、嵌入计算、自适应特征优化、KV缓存认证、推测解码和低维模型头计算，而云端则执行认证的解码器推理、KV缓存管理、令牌验证和高维词汇投影。端点融合部分输出，应用语言自适应掩码并采样目标令牌。所有传输数据和截断的logits均经过量化并使用AES-GCM加密以保护隐私，核心轻量模块、草稿参数和缓存访问策略保留在本地以避免泄露。该框架通过优化的流式处理、批处理和量化的ONNX部署，支持包括纯CPU、配备GPU及嵌入式设备在内的异构设备。评估表明，与基线分割推理相比，该框架将每令牌延迟降低了最多46.1%，并将下行负载降低了最多67.4%，同时保持与完全云端推理相当的性能。

## Abstract
On-device LLM inference faces a trilemma of response latency, limited hardware resources and user privacy. Full cloud inference delivers strong computing power but exposes user prompts and dialogue data, while standalone on-device inference is unfeasible for most consumer and embedded edge devices. This paper presents a privacy-centric edge-cloud collaborative LLM inference framework built on endpoint-authenticated KV cache. Local endpoints handle input preprocessing, embedding computation, adaptive feature optimization, KV cache authentication, speculative decoding and low-dimensional model head calculation, while the cloud conducts authenticated decoder inference, KV cache management, token verification and high-dimensional vocabulary projection. Endpoints fuse partial outputs, apply language-adaptive masking and sample target tokens. All transmitted data and truncated logits are quantized and AES-GCM encrypted for privacy, with core lightweight modules, draft parameters and cache access policies kept local to avoid leakage. The framework supports heterogeneous devices including CPU-only, GPU-equipped and embedded devices via optimized streaming, batching and quantized ONNX deployment. Evaluations demonstrate that the framework reduces per-token latency by up to 46.1\% and downlink payloads by up to 67.4\% over baseline split inference, retaining comparable performance to full cloud inference.

  - **纯云端推理**：算力充足但需要上传原始提示词和对话标识符，带来隐私泄漏和通信开销。
  - **纯本地推理**：对大部分消费级 CPU、嵌入式设备几乎不可行，因为内存与算力不足。
  - **现存分割推理**：一定程度上减少了本地计算，但缺乏对 KV‑Cache 的细粒度控制、缺乏对不同端侧设备的自适应方案、也未将隐藏状态泄漏防护与网络传输安全整合设计。
- **整体含义**
  - 本文旨在**兼顾延迟、带宽效率与隐私保护**，提出一种以端点侧授权 KV‑Cache 为核心的隐私感知边缘‑云协同推理架构，使得端侧保留隐私敏感决策和轻量计算，云侧仅执行被授权的解码计算，最终在异构设备（纯 CPU、GPU、嵌入式）上实现高效可部署的 LLM 推理。

## 2. 方法论
### 2.1 核心思想
- 将**控制权**从云端移交给端侧：端侧明确授权是否以及多少历史 KV‑Cache 条目可用于当前生成步骤；原始文本和完整 logits 分布不上传或下传。
- 将推理拆分为：端侧预处理、特征准备、推测解码、局部 logits 投影；云侧执行授权解码、KV‑Cache 维护、高维 logits 投影和推测验证。
- 通过语言自适应 logits 压缩、设备自适应特征传输和端侧保留的 LoRA 权重降低信息暴露，并以 AES‑GCM 保护网络传输。

### 2.2 关键技术细节
- **设备自适应特征准备**
  - 纯 CPU 端：将初始特征按列分块，使用流式传输，解密和本地计算与接收重叠，降低等待延迟。
  - GPU 端：本地模块按 Transformer 层或注意力头切片，支持多会话批处理与张量并行。
  - 边缘设备：通过量化并导出到 ONNX Runtime 执行，兼容硬件加速后端。
- **云端授权解码器推理**
  - 云端收到特征张量和控制元组 `(flag_Cache, L_Cache, dtype, ℓ)`。
  - 若允许使用缓存，则从历史 KV 条目中选取最多 `L_Cache` 条；否则忽略缓存并仅从当前特征开始解码。
  - 注意力计算中拼接历史 KV 和新投影的 K、V；通过量化与加密传输隐藏状态。
- **分布式词汇投影**
  - 将 LM‑head 权重沿隐藏维度拆分为端侧部分 `W_end`（维度 `D_end`）和云侧部分 `W_cloud`（维度 `D_cloud`），端侧计算 `z_end = h_end W_end`，云侧计算 `z_cloud = h_cloud W_cloud + b`，端侧按元素相加恢复完整 logits。
  - 在语言受限生成时，端侧根据语言标识 `ℓ` 确定词汇子集 `V_ℓ`，云侧仅返回 `z_cloud[V_ℓ]`，端侧融合后对词汇集外维度赋予 −∞，从而大幅限缩下行通信量。
- **推测解码与云端验证**
  - 端侧搭载轻量草稿模块（MLP 或小型 Transformer），基于云返回的隐藏状态预测 `N` 个未来 token。
  - 云侧在授权 KV 窗口下验证草稿 token；接受者提交至云端缓存，拒绝者从最后一个接受位置重新计算，保持缓存与目标模型一致。
- **传输保护与隐藏状态保护**
  - 张量先量化为 INT8，再以 AES‑GCM 加密传输（不同块或消息使用独立 nonce），对网络攻击者实现机密性与完整性保护。
  - 端侧 LoRA 适配的轻量模块、草稿参数以及本地投影权重均不上传，降低可逆攻击风险。
  - 端侧通过 `flag_Cache` 和 `L_Cache` 显式控制历史上下文窗口曝光。

### 2.3 算法流程
- 端侧接收输入 `x`→ 分词得到 `u`→ 计算嵌入并检测语言 `ℓ`，设定缓存控制标志。
- 根据端类型准备特征，与元组一起加密发送给云端。
- 对每个生成步骤 `t`：
  1. 端侧计算 `z_end[V_ℓ]` 和草稿令牌；
  2. 云侧在授权缓存窗口下返回 `z_cloud[V_ℓ]`；
  3. 端侧融合 logits、施加语言掩码并采样 `y_t`；
  4. 端侧将 `y_t` 和草稿令牌发送至云验证；
  5. 云提交已接受令牌，必要时从第一个拒绝令牌重新计算；
  6. 若生成结束则停止。
- 端侧解码完整响应。

## 6. 主要结论与发现
- **延迟全面改善**：相比朴素分割推理，所提框架在 CPU、GPU 和边缘端分别获得 29.4%、46.1% 和 26.8% 的端到端延迟降低，且远优于 Cloud‑only 方案。
- **通信大幅压缩**：中文/英文单一语言模式下，下行 logits 负载相比完整分布分别减少 56.2% 和 66.5%（最高 67.4%）；混合语言场景下无压缩优势，但推测解码和分割投影仍提升效率。
- **组件贡献显著**：推测解码对延迟影响最大；语言自适应压缩与分割词汇投影主要降低下行载荷；CPU 块流式重叠传输对纯 CPU 设备性能提升至关重要。
- **生成质量接近全云水平**：Next‑token agreement 达 98.6%，任务级指标（MMLU、GSM8K、HumanEval）与全云模型差距极小，退化主要来自 8 位量化、语言掩码和局部投影近似。
- **隐私暴露受限**：云侧不接收原始提示、对话 ID、端侧模块权重或完整 logits；但仍会看到解密后的特征张量（为推理所必需）和部分验证元数据，故提供的是系统级暴露缩减而非形式化隐私安全。

## 8. 不足与局限
- **隐私保证非形式化**
  - 仅依赖 AES‑GCM 和本地权重隐蔽，不能抵抗具备辅助知识或可进行模型逆向、成员推断的强攻击者。未在真实攻击（如中间隐藏状态重建实验）下评估隐私泄漏强度。
- **实验场景较为单一**
  - 仅使用单一 7B 模型和固定网络条件；未验证多会话并发、真实波动网络（丢包、拥塞）、超长上下文等挑战下的鲁棒性。
- **训练与适配开销未披露**
  - 缺少 LoRA 模块的训练资源需求、时间及数据信息，草稿模块的训练细节也未公开，可复现性受限。
- **安全性依赖于密钥管理**
  - AES‑GCM 的有效性依赖于端‑云间的密钥协商与保存，且在云侧仍可解密部分必要张量，未考虑云服务商本身被攻陷的场景。
- **部分性能收益未覆盖混合语言场景**
  - 语言自适应压缩在需要多语言混合生成时无效果，推测接受率也可能因语言切换而波动，但文中仅以单语言模式给出主要增益。
