## 摘要
Text-to-SQL 越来越多地部署在数据提供者和用户之间的信任边界上。此类部署必须平衡三个相互竞争的需求：策略合规性、答案覆盖率和成本约束。现有方法通常根据查询中提到的列来决定是否拒绝，并随机执行。然而，查询是否合规不仅取决于涉及哪些列，还取决于如何使用这些列，而随机执行无法确定性地排除违规行为。我们将此需求形式化为基于语义用途的列使用策略：输出、过滤条件和聚合参数。我们通过将每种角色与解码器跟踪的语法产生式对齐来集成该策略。由此得到的系统 PCC-SQL 应用逐 token 的 logits 掩码，在单次解码过程中确定性地消除所支持 SQL 片段上的单查询列使用违规行为。在三个基准测试和三个开源模型上，PCC-SQL 在 Spider-CU 上实现了 0% 的泄漏率和高达 88.7% 的覆盖率，同时 token 数量保持在直接提示的 +10% 以内。我们还通过执行准确率评估了语义对齐。

## Abstract
Text-to-SQL is increasingly deployed across trust boundaries between data providers and users. Such deployment must balance three competing requirements: policy compliance, answer coverage, and bounded cost. Existing approaches typically decide refusal based on which columns a query mentions and enforce it stochastically. Whether a query is compliant, however, depends not only on which columns appear but on how they are used, and stochastic enforcement cannot deterministically rule out violations. We formalize this requirement as a column-use policy over semantic use: output, filter condition, and aggregation argument. We integrate the policy by aligning each role with grammar productions tracked by the decoder. The resulting system, PCC-SQL, applies a per-token logits mask that deterministically eliminates single-query column-use violations on the supported SQL fragment in a single decoding pass. Across three benchmarks and three open-source models, PCC-SQL achieves 0% Leakage Rate and Coverage up to 88.7% on Spider-CU, while staying within +10% tokens of direct prompting. We additionally assess semantic alignment with execution accuracy.

- **研究动机**：需要一种能在生成时刻就防止违规，且能根据列的**使用角色**灵活决策的策略表达与执行机制，从而在单次解码中实现确定性安全、高覆盖和低成本。

## 2. 方法论

### 核心思想
将列级访问控制从“列名是否出现”提升到“**列在 SQL 中的语义角色**”层面，并通过约束解码在每一步 token 生成时剔除违规列，实现**单次解码、确定性的零泄露**。

### 关键技术细节

#### (1) 列使用策略（Column-use policy）与角色定义
- **四种列权限** π：
  - `PUBLIC`：无限制，可出现在任意位置。
  - `CONDITIONONLY`：仅可在 `JOIN … ON`、`WHERE` 或 `HAVING` 中作为条件列，不能直接输出。
  - `AGGREGATEONLY`：只能作为 `COUNT()`, `SUM()`, `AVG()` 的聚合参数，不能出现在 SELECT 输出、WHERE 或 JOIN 条件中。
  - `HIDDEN`：完全不可出现在 SQL 中。
- **四种语法角色** ρ：`SEL`（SELECT/GROUP BY/ORDER BY 的输出列）、`JOIN`（JOIN … ON 中的连接列）、`WHERE`（WHERE/HAVING 中的条件列）、`AGG`（聚合函数参数列）。
- **权限矩阵** `perm(π, ρ)`：定义每种列权限在哪些角色下允许出现。例如 `CONDITIONONLY` 允许在 JOIN 和 WHERE 中出现，但不允许在 SEL 或 AGG 中出现。

#### (2) PCC-SQL 约束解码器
- **两个组件**：
  1. **状态跟踪器（State Tracker）**：在解码过程中维护当前 SQL 语法位置（当前子句、聚合嵌套状态、作用域表等），由此推导当前 token 即将填入的**角色** ρ。
  2. **Logits 处理器（Logits Processor）**：在每一个**列名生成位置**，根据当前角色 ρ 和作用域内的列权限，计算允许的列集合：
     ```
     Allowed = {c ∈ Scope | perm(P(c), ρ) = ⊤}
     ```
     然后将不在允许集合内的列名 token 的 logits 设为 -∞，保证模型只能从合规的列中采样。
- **拒绝路径（Refuse）**：若在任何列名位置上 `Allowed` 为空（即没有合规列可用），解码器直接终止并输出 `REFUSE`。
- **确定性保证**：由于 SQL 前缀可唯一确定列名位置的语义角色，加上上述硬掩码，**所有生成的列名都必然满足权限矩阵**，因此在支持的 SQL 片段上实现了单步解码的零违规。

#### (3) 其他工程细节
- 聚合参数识别通过回扫括号平衡检测，支持 `COUNT(*)` 始终放行。
- 子查询中的列引用通过 SELECT scope 栈管理作用域。
- 列名以 `表.列` 路径形式存储为 trie，并处理分词器 BPE 合并情况。

## 3. 实验设计

### 数据集与基准
- **Spider-CU**（1,034 条）：基于 Spider dev 集构建，加入**四权限**（PUBLIC / CONDITIONONLY / AGGREGATEONLY / HIDDEN）标注。通过正则匹配（PII 类列名）与训练/开发集 SQL 中角色统计自动生成策略。
- **BIRD-CU**（1,534 条）：BIRD dev 集类似扩展，同样采用四权限评估。
- **Spider-ACL**（19,624 条）：来自 Klisura et al. (2026) 的外部基准，采用二值权限（PUBLIC / HIDDEN），通过 GRANT 语句转换得到。用于验证跨策略标注的迁移能力，但不涉及角色依赖权限。

### 评估指标
- **Leakage Rate**（↓）：生成不安全 SQL 的比例（越小越好）。
- **Coverage**（↑）：生成安全 SQL 的比例（包括原本可回答的查询及合规替代答案）。
- **Recall**（↑）：对原 SQL 合规的查询（SQL-gold）返回安全并执行的准确结果的比例。
- **Recovery Rate**（↑）：对原 SQL 违规但存在合规替代方案的查询（Refuse-gold）返回安全 SQL 的比例。
- **Failure Rate**（↓）、**Calls/query**、**Tokens/query**、**Tokens/Safe SQL Return** 等成本指标。
- **执行准确度**：在 SQL-gold 查询上，预测 SQL 执行结果与 gold SQL 执行结果完全一致的比率。

### 对比方法
- **Direct Prompting**：零样本直接生成，无任何防御。
- **Retry (N=3)**：检测违规后重试最多 3 次，违规时返回 REFUSE。
- **2-step Verifier** (Klisura et al. 2026)：生成后用同一模型验证 SQL 是否合规，不合规则回退到 REFUSE。
- **IterGen + role check**：基于 grammar 的迭代生成，在 SQL 标识符边界进行角色检查与策略对比，失败则回滚重采样。
- **PCC‑SQL**（本文方法）。

### 模型
- 开源：Qwen 2.5 Coder 7B / 32B Instruct，DeepSeek Coder 6.7B Instruct（用于所有方法，包括 PCC-SQL）。
- API 模型：Claude Haiku 4.5、Opus 4.5（仅用于基准方法，因 PCC-SQL 需 logits 干预）。

## 6. 主要结论与发现

- **PCC-SQL 是唯一在所有开源模型和基准上实现 0% Leakage Rate 的方法**，且不需要事后验证或重试。
- **覆盖率显著优于基线**：在 Spider-CU 上，Qwen 7B/32B 达到大约 80%–88% 的 Coverage，特别是在 Refuse-gold 查询的 Recovery Rate 上，领先最强基线（Retry）超过 74 个百分点。
- **语义正确性未牺牲**：执行准确度表明，PCC-SQL 的高 Coverage 并非源自输出语义无关的 SQL；在某些模型-基准组合上甚至优于部分基线。
- **成本极低**：PCC-SQL 保持 1 次 LLM 调用，且每查询 token 数只比直接提示高不到 10%；而重试或两步验证方法在违规频繁时吞吐量翻倍。
- **约束解码的覆盖能力依赖基础模型能力**：DeepSeek Coder 6.7B 的 Recovery Rate 明显低于 Qwen 系列，说明模型自身的 SQL 规划能力影响合规替代 SQL 的生成成功率，但**零泄漏保障不受模型能力影响**。
- **对策略分布扰动稳健**：当构造基准的概率参数改变时，PCC-SQL 的泄漏率保持 0%，Coverage 仅小幅波动，与基线的优势关系不变。

## 7. 论文亮点

- **语义级别的访问控制**：首次将列使用角色引入列级策略，突破了传统二值（出现/不出现）的局限，能够支持“仅可聚合”“仅可条件”等精细查询。
- **生成时刻确定性安全**：通过将策略编码为 logits 硬掩码，在解码过程中 100% 防止违规列出现，无需依赖生成后验证，这是同类工作尚未实现的目标。
- **兼顾安全与可用性**：同时实现 0% 泄露、高覆盖率和高执行效率，证明安全与可服务性不必互相妥协。
- **系统简洁高效**：实现为轻量级 LogitsProcessor，无需额外模型调用或训练，易于集成到现有 LLM 推理管线。
- **全面且公平的评估体系**：在两个自家构建的四权限基准和一个外部二值基准上对比了多种主流方法，进行了多维度指标和开销测量，并附带消融和语义正确性验证，结论令人信服。

## 8. 不足与局限

- **SQL 片段受限**：当前仅支持基础 SELECT/JOIN/WHERE/HAVING/聚合，**不支持集合操作、CTE、窗口函数、CASE 表达式等复杂语法**。对于实际复杂查询，覆盖范围受到一定制约。
- **角色定义相对固定**：仅区分为 SEL、JOIN、WHERE、AGG 四种角色，对于更细粒度的需求（如区分 GROUP BY 与 SELECT 输出、ORDER BY 等）尚未延伸。
- **策略标注为规则合成**：Spider-CU 和 BIRD-CU 的策略是通过正则和 SQL 统计自动生成的，仅经过小规模人工抽查，可能无法完全代表真实场景的访问控制需求；Spider-ACL 虽来自外部但也是基于 GRANT 的二值权限，并未验证人类定制的**角色敏感策略**。
- **未能处理多查询推理攻击**：安全保证限定在单查询的列使用合规性，无法应对通过多次查询联合推断敏感信息的攻击，也没有提供 k-匿名、差分隐私等隐私保护。
- **拒绝路径可能过于保守**：当所有安全列均被掩码时，系统直接返回 REFUSE，即使可能通过重写查询结构（如引入子查询或 CASE）获得安全形式，这会在基础模型能力较弱时导致额外的覆盖率损失（假阴性）。
- **仅适用于白盒模型**：PCC-SQL 依赖 logits 级别的干预，只能用于开源模型，无法直接迁移到 API 黑盒模型。
- **执行准确度下降风险**：虽然整体安全 SQL 的执行匹配度未严重下滑，但在部分模型（如 Qwen 7B、DeepSeek 6.7B）上仍落后最优基线 5-18 个百分点，说明硬约束可能会在一定程度上影响生成质量，存在“安全-质量”折衷。
