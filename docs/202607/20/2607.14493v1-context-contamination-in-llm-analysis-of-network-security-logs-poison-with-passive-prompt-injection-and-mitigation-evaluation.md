## 摘要
大型语言模型（LLM）越来越多地部署在安全运营中心，用于日志分析任务，包括摘要生成、告警分类和威胁调查。这些系统从面向外部的服务中获取日志，并将网络日志作为自然语言上下文进行处理，以生成安全洞察。我们证明，这种架构模式引入了一个严重漏洞：攻击者可以在日志生成字段中嵌入提示注入载荷，这些载荷持久存储在系统中，当分析师查询LLM时即被执行，从而实现我们称之为被动提示注入的攻击。我们提出了LogInject，一个系统性评估此类威胁的框架。利用LogInject-1.0（包含12,847条日志条目、其中2,569个对抗样本的基准），我们在四个攻击目标（活动隐藏、误报生成、信息窃取和输出劫持）上评估了三个生产级LLM。结果显示，在基线条件下，攻击成功率高达88.2%（模型间平均83.4%）。我们引入了上下文拼接技术，将载荷分散在多个日志条目中，以逃避无状态过滤器并利用LLM的长上下文推理，实现了76.4%的成功率。作为缓解措施，我们评估了结合输入过滤、提示强化和输出验证的分层防御，证明可减少90.4%的攻击，尽管仍有8.4%的残余漏洞。我们的结果揭示，基于LLM的日志分析产生了一种固有的混淆代理人漏洞，不可信数据与可信指令无法区分地竞争模型注意力，因此需要纵深防御架构和对安全关键决策的持续人工监督。

## Abstract
Large Language Models are increasingly deployed in Security Operations Centers for log analysis tasks including summarization, alert triage, and threat investigation. These systems ingest logs from external-facing services and process network logs as natural language contexts to generate security insights. We demonstrate that this architectural pattern introduces a critical vulnerability: adversaries can embed prompt injection payloads in log-generating fields that persist in storage and are executed when analysts query the LLM, achieving what we term passive prompt injection. We present LogInject, a systematic framework for evaluating these threats. Using LogInject-1.0, a benchmark of 12,847 log entries including 2,569 adversarial samples, we evaluate three production LLMs across four attack objectives: activity concealment, false positive generation, information exfiltration, and output hijacking. Our findings reveal an up to 88.2% attack success rate (83.4% average across models) under the baseline conditions. We introduce Context Stitching, a novel technique that fragments payloads across multiple log entries to evade stateless filters while exploiting LLM long-context reasoning, achieving a 76.4% success rate. As mitigation, we evaluate layered defenses by combining input filtering, prompt hardening, and output validation, demonstrating a 90.4% attack reduction, although 8.4% residual vulnerability persists. Our results establish that LLM-based log analysis creates an inherent confused deputy vulnerability where untrusted data and trusted instructions compete indistinguishably for model attention, requiring defense in-depth architectures and continued human oversight for security-critical decisions.

- **核心问题**：安全运营中心（SOC）日益依赖大语言模型（LLM）对网络日志进行自动摘要、告警分类和威胁调查。然而，LLM在处理由外部服务生成的日志时，无法在架构上区分“数据”与“指令”，导致攻击者可在HTTP头、SSH用户名、JSON字段等可控字段中嵌入恶意载荷（即**被动提示注入**）。这些载荷在日志存储中持久保存，直到分析师触发LLM分析时才被激活，从而污染分析结果。
- **整体含义**：这种“上下文污染”（Context Contamination）是一种**固有的混淆代理人漏洞**——LLM的Transformer自注意力机制对所有标记一视同仁，无论它们是来自系统提示还是来自不受信任的日志数据。其危害包括：隐藏恶意活动、制造虚假告警、泄漏系统提示或敏感信息、劫持模型输出等。论文旨在**系统性地量化该威胁的严重性**，并提供防御策略评估，而非概念验证。

## 2. 论文提出的方法论：核心思想、关键技术细节
- **LogInject 评估框架**：将威胁模型操作化，支持可复现的对抗评估。
  - **攻击复杂性分级**（三个等级）：
    1. **原子注入（L1）**：完整载荷置于单一日志字段，利用伪造对话标记（如 `SYSTEM:`）覆盖系统指令。
    2. **上下文拼接（L2, Context Stitching）**：将载荷拆分为多个日志条目，每个条目单独看无害，但当LLM在批次中同时处理这些条目时，载荷被重组并执行。该技术利用了**无状态过滤器的缺陷**——每个片段可通过WAF/签名检测，但LLM的长上下文理解能力会把它们“缝”成完整攻击。
    3. **混淆规避（L3）**：通过插入空字符、Base64/十六进制编码、多语言同形字等方式隐藏关键字，利用LLM的纠错和指令跟随能力还原载荷。
  - **攻击目标分类**：
    - **OBJ‑CONCEAL**：隐藏恶意活动，要求模型将可疑日志总结为“无异常”。

    - **OBJ‑EXFIL**：在输出中附带系统提示等秘密信息（例如通过URL外泄）。
    - **OBJ‑INSTRUCT**：劫持输出内容，注入钓鱼链接或伪装的指令。
  - **防御层级**：评估了三层防御策略及其组合——
    - **输入过滤**：基于正则表达式的关键词封堵（如 `SYSTEM:`、`[INST]`等）。

## 6. 论文的主要结论与发现
- **基线高脆弱性**：在无防御条件下，攻击成功率平均达83.4%（GPT‑4o: 87.3%, Llama‑3‑70B: 88.2%, Claude 3.5 Sonnet: 74.8%），其中活动隐藏（OBJ‑CONCEAL）的平均ASR最高（89.2%），信息窃取最低（78.4%）。
- **攻击复杂度梯度**：ASR随技术升级而降低，但即使是最复杂的L3混淆攻击仍保持66.7%的成功率。
- **上下文拼接可行性**：即使将载荷拆分为10个碎片，成功率仍可超过50%（GPT‑4o和Llama‑3‑70B），说明无状态检测架构基本无效。碎片在同一批次中时的ASR（81.4%）远高于跨批次（34.2%），提醒防守方可缩短批窗口作为妥协。
- **防御有效性**：
  - 单一防御效果有限：输入过滤仅减少10%的ASR；输出验证同样有限；Spotlighting能将ASR从87.3%降至51.4%，是**最有效的单项防御**。
  - 分层防御（输入过滤 + Spotlighting + 输出验证）可将ASR降至8.4%，相对降幅90.4%，但仍有不可忽视的残余风险。
  - Spotlighting的有效性随上下文长度增加而显著下降（Lost‑in‑the‑Middle现象），16K tokens时ASR回升至70%以上。
- **根本原因**：Transformer架构中控制平面与数据平面的”塌缩“使得任何基于静态过滤的方案都无法根治；语义消毒在理论上不可判定，攻击者可利用LLM的指令跟随与纠错能力绕过防御。
- **运营影响**：组合防御会带来3.4%的良性准确率下降（从94.2%到90.8%），估计在中等SOC中每次值班将增加约28个分析师工时，建议采用格式感知Prompt来降低误报。
