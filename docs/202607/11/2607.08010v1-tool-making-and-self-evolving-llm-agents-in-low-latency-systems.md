## 摘要
生产环境中的LLM智能体通常会在每次请求时为相同的程序化步骤重新生成代码，这既增加了延迟，也降低了可靠性。我们用一条智能体工具制造流水线替代了这一推理时编码循环，该流水线在部署前将重复的标准操作程序（SOP）步骤编译为经过验证且带有版本管理的工具。工具制造器在实时环境中进行合成：它收集执行轨迹，观察后端模式和具体数值，生成候选工具，并根据带标签的案例进行修复。在运行时，生产智能体直接调用这些工具，仅在需要时才回退到代码生成。我们将该方法部署在一个订单履行中心的告警分类系统中，在该系统中，智能体根据一个包含44个节点的SOP，在多种异构指标后端上诊断告警。在生产环境中，工具调用将p50延迟降低了42%。在1,500个历史告警上，通过抑制重复步骤中逐次执行的差异，它们将端到端错误率最多降低了53%。由于工具返回紧凑的结构化判定结果，它们还支持一种更简单的直接调用架构，在对照消融实验中进一步将p50延迟降低了62%。版本化的工具还提高了可审计性，并暴露了规范差距和上游数据漂移。我们的结果表明，自进化的智能体可以让工业LLM系统变得更快、更可靠且更易于运维。

## Abstract
Production LLM agents often waste latency and reliability by regenerating code for the same procedural steps on every request. We replace this inference-time coding loop with an agentic tool-making pipeline that compiles repeated SOP steps into validated, versioned tools before deployment. The tool-maker grounds synthesis in the live environment as it collects execution traces, observes backend schemas and values, generates candidate tools, and repairs them against labeled cases. At runtime, the production agent calls these tools directly and falls back to code generation only when needed. We deploy the approach in a Fulfillment Center alarm-triage system, where an agent diagnoses alarms against a 44-node SOP over heterogeneous metric backends. In production, tool calls reduce p50 latency by 42%. On 1,500 historical alarms, they reduce end-to-end error rate by up to 53% by suppressing run-to-run variance in repeated steps. Because tools return compact structured verdicts, they also enable a simpler direct-call architecture, reducing p50 latency by a further 62% in a controlled ablation. Versioned tools also improve auditability and expose specification gaps and upstream data drift. Our results show that self-evolving agents can make industrial LLM systems faster, more reliable, and easier to operate.

- **核心问题**：当前工业界部署的LLM编码智能体（如CodeAct范式）在每一次请求推理时，都会为相同的标准操作程序（SOP）步骤重新生成并执行代码。这种“推理时重复编码”导致了三个主要弊端：
  - **高延迟**：多次LLM调用和代码生成增加响应时间。
  - **不可靠**：即使是重复任务，每次运行也存在方差，可能偏离规范。
  - **高成本**：大量重复的计算和令牌开销。
- **整体含义**：本文提出了一种“自进化智能体”模式，通过**离线工具制造流水线**，将重复的SOP步骤预先编译为经过环境验证、带版本管理的确定性工具。运行时直接调用这些工具，仅在失败时回退到代码生成，从而在保留灵活性的同时，使工业LLM系统更快、更可靠、更容易审计和运维。

## 2. 论文提出的方法论
- **核心思想**：将SOP决策树中的每个节点（观察、根因、约束）编译成一个独立的可调用工具函数。该工具在离线阶段通过“数据接地”与“测试-修复”循环生成，运行时由主智能体或子智能体直接调用，返回结构化判定结果。
- **关键技术细节与流程**：
  - **数据收集子智能体**：针对每个节点，在3个带标签的训练案例上运行基线CodeAct子智能体，收集完整的执行轨迹（含查询代码、后端响应、观察到的模式、判定是否与标签一致）。该轨迹为工具制造器提供了SOP文本中缺失的执行细节（字段名、数据类型、值域等）。
  - **工具制造器LLM**：根据SOP节点文本、节点在决策树中的位置以及数据收集轨迹，生成候选工具函数。
  - **测试-修复循环**：
    - 将候选工具在所有训练案例上测试，对失败案例，由**反射器LLM**生成诊断说明。
    - 工具制造器根据诊断与失败案例重写工具。
    - 循环最多3轮，最终选择在训练集上通过率最高的候选工具部署。
  - **运行时调用**：工具接收统一签名（仓库、时间戳、上下文），直接返回判定（true/false/no data）及观察到值、阈值、解释文本。出现异常时回退到传统CodeAct子智能体。
  - **智能体微调**：从强教师模型蒸馏，用混合轨迹（75%轨迹部分工具随机不可用，25%轨迹全部无工具）微调学生模型，确保工具不可用时的回退能力。
