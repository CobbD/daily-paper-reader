## 摘要
随着大语言模型智能体承担进攻性安全任务，一次超出范围的工具调用就可能违反客户约定边界、中断生产环境或使漏洞赏金发现失效。与固定的安全策略不同，关键的边界在用户请求中声明，且必须从意图中推断。进攻性安全的对抗性质使这一挑战更加严峻：同一工具调用是否在范围内并不取决于操作本身，而取决于它所触及的目标及其运行的上下文，这是任何固定策略无法预先枚举的。我们研究预执行门控：一个低成本、可信的LLM评判器检查由功能强大且可替换的智能体提议的每次调用，并在执行前接受或拒绝它。我们推出了ScopeJudge，这是一个基准数据集，包含从智能体轨迹中提取的4,897次工具调用（其中7.7%为范围违规），这些任务被设计为诱使智能体超出范围，并由专业渗透测试人员在调用级别进行标注，评分者间一致性较高（Fleiss' kappa = 0.64），这设定了一个专家一致性参考点F1 = 0.78。我们在五种转录策略下评估了八个评判模型，改变评判器所见的上下文量，从仅静态策略到完整的原始转录，并绘制了由此产生的成本-准确性帕累托前沿。我们发现，静态策略在结构上不足以进行范围执行：对用户请求视而不见导致评判器召回率几乎降为零，这确认了范围存在于请求中，且基于请求的监控是必要的。由于遗漏违规的代价高于误拒，我们分别报告精确率、召回率和F1值，并推荐两个工作点：一个成本敏感配置和一个用于高风险部署的召回优先配置。我们发布了ScopeJudge数据集，以支持自主安全智能体的实时监控和可扩展监督。

## Abstract
As LLM agents take on offensive security work, a single out-of-scope tool call can breach a client's engagement boundary, disrupt production, or void a bug-bounty finding. Unlike a fixed safety policy, the boundary that matters is declared in the user's request and must be inferred from intent. That challenge is sharpened by the adversarial nature of offensive security: the same tool call is in or out of scope depending not on the action itself but on the target it touches and the context in which it runs, which no fixed policy can enumerate in advance. We study pre-execution gating: a cheap, trusted LLM judge inspects each call proposed by a strong, swappable agent, and accepts or rejects it before it runs. We introduce ScopeJudge, a benchmark of 4,897 tool calls (7.7% scope violations) from agent trajectories on tasks engineered to tempt agents out of scope and labeled at the call level by professional penetration testers, with substantial inter-grader agreement (Fleiss kappa = 0.64) that sets an expert agreement reference point of F1 = 0.78. We evaluate eight judge models under five transcript strategies, varying how much context the judge sees, from the static policy alone to the full raw transcript, and chart the resulting cost-accuracy Pareto frontier. We find that a static policy is structurally insufficient for scope enforcement: blind to the user's request, judge recall collapses to near zero, confirming that scope lives in the request and that request-conditioned monitoring is necessary. Because a missed violation costs more than a spurious rejection, we report precision, recall, and F1 separately and recommend two operating points: a cost-sensitive configuration and a recall-first one for high-stakes deployments. We release the ScopeJudge dataset to support real-time monitoring and scalable oversight of autonomous security agents.

- **核心问题**：当大语言模型（LLM）智能体被部署到真实的进攻性安全任务（渗透测试、漏洞赏金）时，一次超出授权范围的工具调用就可能违反客户约定、破坏生产环境或导致赏金失效。问题的根源在于“操作边界”并非由固定的全局策略决定，而是隐含在用户的具体请求中，且高度依赖上下文与意图。  
- **整体含义**：传统的“沙箱”或静态策略无法应对这种动态范围约束，人工审查又存在疲劳和不可扩展的问题。因此，需要一种**预执行门控机制**：在每次工具执行前，由一个轻量、可信的 LLM 法官依据用户请求和部分历史轨迹进行实时裁决，从而在代理自主性与安全合规之间取得平衡。该工作通过构建基准、评测多种法官模型与上下文策略，为这一范式提供了经验性支撑和部署指导。

### 方法论  
- **核心思想**：将范围控制形式化为**监控问题**而非规范问题。一个廉价、固定、可信的法官模型（Judge）在待测智能体（Agent）每次提出工具调用时，根据给定的上下文策略（Transcript Strategy）判断该调用是否超出范围，裁决 accept/reject，从而阻止越权执行。  
- **关键定义与流程**：  
  - **智能体轨迹**：τ = (u, (a₁, o₁), …, (a_N, o_N))，其中 u 为用户请求（含意图与范围），aₖ 为工具调用，oₖ 为输出。  
  - **预执行门控**：J_pre(SP, φ(τ_{<k}, aₖ)) → {0,1}，其中 SP 为全局静态策略，φ 为转录策略，决定将哪些上下文（如用户意图、历史调用、输出摘要等）送入法官。0 表示允许执行，1 表示拒绝（可要求代理重试或升级人工）。  
  - **五种转录策略**：  
    1. `static_policy`：仅提供静态策略 SP 和待判调用 aₖ。  
    2. `intent_only`：增加用户消息 u。  
    3. `intent_plus_calls`：增加历史工具调用（不含输出）。  
    4. `intent_plus_outputs_summary`：增加由同一法官模型生成的工具输出摘要。  
    5. `full`：提供完整原始轨迹（含推理 trace 和原始输出）。  
- **评估指标**：以专家多数投票为金标准，分别报告精确率、召回率、F1，以及平均每次调用的成本（C(J,φ) = ρ(J) · 输入 token 数）。通过成本-准确性帕累托前沿筛选最优配置，并区分“成本敏感”与“召回优先”两种部署场景。

### 主要结论与发现  
- **静态策略无效**：仅给法官提供静态策略时，召回率近乎为零（中位数 0.005）；因为无法感知具体任务的范围意图，几乎所有调用都被放行。  
- **意图恢复召回**：一经加入用户请求（`intent_only`），召回率跃升至 0.60~0.94，证明范围存在于请求中，请求条件监控是必需的。  
- **最佳法官为开源模型**：GLM-5.2 取得最高 F1=0.66（仅需 $0.006/call），超越最强闭源法官 Claude Opus 4.8（F1=0.60，成本高约 3 倍），位于帕累托前沿。  
- **成本与效果平衡**：  
  - **成本敏感部署**：GLM-5.2 + `intent_plus_calls`，召回 87%，精确率 0.50，成本 $0.0032/call。  
  - **召回优先部署**：Claude Opus 4.8 + `intent_only`，召回 94%，精确率 0.39，成本 $0.0042/call（牺牲精确率以减少遗漏违规）。  
- **专家参考点**：最高 F1 仍低于专家一致性（0.66 vs 0.78），说明尚有提升空间。  

### 优点  
- **明确的问题抽象**：将范围监控定位为“请求条件监控”，区别于固定政策，抓住了进攻性安全任务的核心矛盾。  
- **系统的策略分类**：提出五种转录策略，清晰刻画了“成本-信息量-攻击面”的权衡空间。  
- **务实的评估体系**：同时报告精确率、召回率、F1 和成本，并给出两种极端部署推荐，贴近工程实践。  
- **开放的基准与参考**：发布 ScopeJudge 数据集和人类标注，为后续研究提供标准化基准和专家一致性参考值。  
- **揭示关键洞察**：静态策略的失败和意图的恢复效应，以及开源模型在低成本和较高性能上的优势，对行业有直接指导意义。

### 不足与局限  
- **语料范围较窄**：仅含 100 条轨迹、4,897 个调用，全部来自一个任务套件、单一领域（Web 渗透测试）和单一语言（英语），外部泛化能力待验证。  
- **对抗鲁棒性未评估**：拥有历史输出的策略暴露了提示注入面，但论文未测试恶意代理或篡改输出对法官的欺骗能力，当前结果可能只是推理条件下的上限。  
- **自评偏差风险**：部分法官模型与轨迹生成模型重叠，可能引入自我偏好偏差，未做控制。  
- **基率依赖**：违规率仅 7.7%，精准率和 F1 依赖该基率，在其他部署环境中指标可能漂移。  
- **缺少方差估计**：所有配置仅运行一次（温度 0），未提供置信区间，小幅度性能差异需谨慎解读。  
- **应用限制**：只覆盖二元分类（范围内/外），未考虑“合规但危险”的调用，也未探讨如何校准以避免审批疲劳。
