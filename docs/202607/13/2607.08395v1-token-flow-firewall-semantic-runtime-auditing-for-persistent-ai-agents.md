## 摘要
持久AI代理将大语言模型（LLM）从单轮交互扩展为长期运行的软件系统。不同于传统的聊天助手，这些代理中的不安全内容可能通过持久状态、可复用技能和工具介导的交互进行传播，从而形成显著更大的语义攻击面。我们观察到，此类代理中大多数安全关键的交互是通过自然语言token流传递的，包括内存更新、工具参数、检索的文件以及组件间通信。这一观察使一种新的安全范式成为可能：不安全行为可以在到达特权运行时汇点之前作为有风险的语义流被拦截。基于这一见解，我们提出了TokenWall，一个充当代理token流语义防火墙的运行时防御框架。TokenWall对这些流执行边界感知的语义审计，构建结构化的源-汇审计记录，在执行前进行轻量级本地检查，并选择性地将模糊的高风险用例升级到更强的仲裁模块。不同于依赖稀疏审计或远程大型模型监督的先前方法，TokenWall实现了全覆盖的执行前中介，同时减少了远程仲裁和延迟。在CIK-Bench上的实验表明，TokenWall将攻击成功率降低至12.5%，同时在无需人工确认的情况下保持97.4%的良性可执行通过率。此外，TokenWall在良性用例上仅引入0.69秒的额外延迟，证明语义运行时遏制能够为持久AI代理实现安全与效用的实际权衡。

## Abstract
Persistent AI agents extend large language models (LLMs) beyond single-turn interaction into long-lived software systems. Unlike traditional chat assistants, unsafe content in these agents can propagate through persistent state, reusable skills, and tool-mediated interactions, creating a substantially larger semantic attack surface. We observe that most security-critical interactions in such agents are transmitted through natural-language token flows, including memory updates, tool arguments, retrieved files, and inter-component communications. This observation enables a new security formulation: unsafe behavior can be intercepted as risky semantic flows before reaching privileged runtime sinks. Based on this insight, we propose TokenWall, a runtime defense framework that acts as a semantic firewall over agent token flows. TokenWall performs boundary-aware semantic auditing over these flows, constructing structured source-sink audit records, applying lightweight local inspection before execution, and selectively escalating ambiguous high-risk cases to stronger arbitration modules. Unlike prior approaches that rely on sparse auditing or remote large-model oversight, TokenWall enables full-coverage pre-execution mediation while reducing remote arbitration and latency. Experiments on CIK-Bench show that TokenWall reduces attack success rate to 12.5% while maintaining a 97.4% benign executable pass rate without human confirmation. TokenWall further introduces only 0.69 seconds of additional latency on benign cases, demonstrating that semantic runtime containment can achieve a practical security-utility trade-off for persistent AI agents.

## 2. 方法论：TokenWall 框架

### 2.1 核心思想
将代理的每一次安全相关交互建模为**token流**（token flow），在流到达受保护的“汇点”前进行**边界感知审计**，并采取允许、重写、推迟或阻止等执行动作。

### 2.2 Token流抽象
一个token流形式化为：  
`f = (x, s, t, c, b)`  
- `x`：载荷（分段为秘密、路径、收件人、指令、工具参数等）  
- `s`/`t`：源/汇（如用户输入、文件、内存→文件写入、工具调用、内存更新等）  
- `c`：运行时元数据（信任级别、权限、会话范围等）  
- `b`：穿越的边界类型  

三种安全表面：  
- **上下文流**：内容进入推理上下文、记忆、技能等  
- **权限流**：可能改变身份、会话绑定、授权状态的转移  
- **能力流**：触发工具执行、文件操作、API请求或对外泄露

### 2.3 分层审计管道
框架由三层构成：

1. **确定性预检查（Precheck）**  
   基于硬编码规则过滤明显违规（如私有端点与执行面桥接、破坏性操作无授权），输出证据标注供后续使用。

2. **本地小模型审计（Small Auditor）**  
   - 接收结构化审计记录（含表面类型、源/汇元数据、载荷分段、预检查证据、所有者意图摘要）。  
   - 输出结构化报告：
     ```
     (d, x', ρ, u, ξ, ι, T, κ, q, e)
     ```
     其中 `d` ∈ {allow, rewrite_and_continue, defer_to_human, block}，`x'`为可选重写载荷，`ρ, u, ξ, ι`分别为风险、不确定性、可利用性、影响评分，`T`为残余风险类型，`κ`为重写完整性，`q`为是否需要大模型复审标志，`e`为证据。  
   - 重写操作允许删除或遮蔽不安全片段，但禁止引入新事实、扩展权限或改变用户目标。若重写无效，按**失败关闭**处理。

3. **回退仲裁（Fallback Arbitration）**  
   当本地审计判定为高影响、高不确定性或不完全重写时，调用更大规模模型（Qwen3.6-Plus）进行最终裁决，动作空间相同。提升条件由表面特定谓词和手动设定的阈值控制。

整体流程见 Algorithm 1（论文附录），确保**预检查‑本地审计‑选择性仲裁**的级联决策。

### 2.4 关键设计选择
- **边界感知**：审计在转移发生前的最后安全点进行，避免状态污染。  
- **结构化作答**：小模型输出包含多维度评分，支撑下游策略（如升级决策）。  
- **选择性升级**：大部分案例由本地小模型直接裁决，仅高不确定性或高风险案例送交远端仲裁，平衡延迟与安全性。  
- **保持良性功用**：通过重写机制，允许在不中断任务的前提下清除危险片段。

## 6. 主要结论与发现

1. **攻击容纳能力**：TokenWall将整体ASR降至12.5%，优于最强基线ClawKeeper（14.7%）及其他所有方法，在上下文、权限和能力三个表面均取得领先。

3. **良性行为保持**：良性通过率97.4%，且**未导致额外的自动阻拦或人工审批**，在低延迟下自行处理常规所有者更新。
4. **设计有效性**：
   - 分层架构必不可少：移除本地审计器导致大模型调用率和延迟骤增，移除回退仲裁导致ASR上升；重写机制有效减少了不必要的升级。

   - 本地审计器可缩放至4B参数获得最佳性价比，更大模型收益递减。
5. **核心洞察**：在信息流层面（token流）进行语义运行时审计，可在不增加过度开销下实现实用的安全‑效用权衡。

## 7. 优点

- **新颖防御视角**：首次将持久代理安全重新表述为**token流的中介**问题，提出前摄性语义防火墙概念，比传统动作级防御更精确。
- **高效分层设计**：预检查‑小模型审计‑选择性仲裁流水线，确保常规案例即时处理、疑难案例深度审查，兼顾安全与延迟。
- **可恢复的重写策略**：支持删除不安全片段而不阻断整体任务，避免简单粗暴的拒绝，提升用户体验。
- **结构化裁决信号**：多维评分（风险、不确定、可利用性等）支持细粒度决策和策略定制。
- **实验扎实**：比较对象广泛，消融分析深入，提供了丰富的诊断指标，揭示了各模块的贡献。
- **低开销**：额外延迟极小（0.69s），无需人工频繁介入，便于实际部署。

## 8. 不足与局限

- **评测范围窄**：仅基于CIK-Bench（一个OpenClaw风格代理的特定基准），未在其他代理框架（如Multi-Agent、企业级工具链）或真实世界工作负载下验证。
- **依赖本地模型质量**：小模型审计器的性能直接决定防御效果，可能存在对抗样本绕过风险；尤其泛化到未见过的攻击模式时可能不可靠。
- **升级仲裁仍需强模型**：尽管调用了大模型仲裁器的比例低，但那些高安全风险案例仍无法完全避免对远程强模型的依赖，可能成为瓶颈或隐私出口。
- **信号有限**：框架严重依赖运行时元数据和授权信号。若代理环境不能提供充分的源/汇信任标注，审计依据将减弱。
- **非完整安全方案**：声明不防御主机受损、凭据外泄、DoS攻击或用户执意授权有害行为的情况，仅聚焦运行时的语义流遏制。
- **延迟实现相关**：报告中延迟依赖具体推理引擎和服务架构，实际部署中可能因模型服务化、缓存等因素产生较大波动。
