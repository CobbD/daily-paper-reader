## 摘要
针对提示注入攻击提供安全保证的防御措施依赖于可信指令与不可信数据之间的严格隔离。在基于文本的环境中，如工具使用API，这种分离自然出现：代理可以根据接口定义进行推理，而无需处理不可信内容。将这些保证扩展到Web代理面临一个根本性挑战：为了感知并与其环境交互，Web代理必须首先观察渲染后的页面，而该页面将可信内容与不可信内容混杂在一起。这种结构上的纠缠消除了安全保证所依赖的信任边界，从而削弱了Web代理的可证明防御。在本文中，我们提出了不可信内容掩蔽（UCM），一种简单而有效的方法，用于在Web环境中恢复这一边界。我们利用了一个关键的结构性洞见：网页的文档对象模型（DOM）编码了足够的信息，可以在不读取其内容的情况下区分可信与不可信区域。我们的框架利用这一点，在不可信区域到达代理之前将其遮蔽，并通过具有严格权限分离的沙盒化接口路由交互，从而使代理能够在观察环境并与之交互的同时，保持与对抗性内容的隔离。代码已公开可用。

## Abstract
Defenses that provide security guarantees against prompt injection attacks rely on strict isolation between trusted instructions and untrusted data. In text-based environments such as tool-use APIs, this separation arises naturally: agents can reason from interface definitions without ever processing untrusted content. Extending these guarantees to web agents faces a fundamental challenge: to perceive and interact with their environment, web agents must first observe the rendered page, which intermingles trusted content with untrusted content. This structural entanglement removes the trust boundary on which security guarantees depend, undermining provable defenses for web agents. In this paper, we present Untrusted Content Masking (UCM), a simple and effective approach that restores this boundary in web environments. We leverage a key structural insight: a webpage's Document Object Model (DOM) encodes sufficient information to distinguish trusted from untrusted regions without reading their content. Our framework exploits this by redacting untrusted regions before they reach the agent and routing interaction through a sandboxed interface with strict privilege separation, thereby enabling agents to observe and interact with their environment while remaining isolated from adversarial content. The code is publicly available.

## 1. 论文的核心问题与整体含义

- **核心问题**：Web 代理在自主浏览和执行任务（如订票、管理邮件）时，网页中嵌入的不可信内容（如用户评论、广告、第三方内容）可能包含恶意提示注入（prompt injection），操纵代理行为，造成安全风险。
- **现有防御的不足**：可证明安全的防御（如指令-数据隔离）在文本API环境中易于实现，但在网页环境中，渲染页面将可信与不可信内容混杂，破坏了隔离所需的信任边界，使这类防御失效。启发式防御则缺乏形式化保证，易被自适应攻击绕过。
- **本文动机与整体含义**：恢复 Web 环境中的信任边界，在不牺牲代理感知和交互能力的前提下，提供对抗提示注入的可证明安全保证。

## 2. 方法论：核心思想、关键技术细节与流程

### 2.1 核心思想
- 利用网页的 **DOM 结构** 在**不读取内容**的情况下区分可信区域与不可信区域。
- 将不可信区域替换为**带标签的占位符**（如 `id: product-review-1`），使代理只看到占位符而无法接触到实际的对抗性内容。
- 当任务**必须读取不可信内容**时，通过一个**隔离模型（Q-Model）** 进行类型受限的查询，确保输出不能携带自由形式的恶意指令。

### 2.2 关键技术细节
- **不可信内容掩蔽（UCM）**：
  - 网站所有者在 DOM 元素上标记 `data-untrusted="true"` 属性，或通过自动推断生成 CSS 选择器。
  - 浏览器侧框架在页面加载时，隐藏不可信元素内容，用包含唯一标识符（如 `qllm_id`）的占位符代替。
  - 动态内容通过 `MutationObserver` 监控，自动处理新插入的不可信元素。
- **沙箱化交互（Quarantined Model）**：
  - 代理调用 `quarantined_llm_analysis` 工具，指定目标元素 ID、自然语言问题、返回类型（`bool`、`int`、`float`、`date`、`enum`）及类型约束。
  - Q-Model 是独立的 LLM 实例，只接收被查询元素的文本内容，返回**结构化答案**（必须能解析为指定类型），从而阻止注入指令随自由文本传播回主代理。
  - 严格的系统提示限制 Q-Model 的行为；主代理对每个元素的查询次数也有限制（最多5次）。
- **自动化边界推断**（针对被动诚实所有者）：
  - 使用 LLM 分析**内容净化后的 DOM**（所有文本替换为长度占位符，图片和链接匿名化）。
  - 通过迭代提示（三轮），LLM 输出 CSS 选择器来定位不可信区域。
  - 该方法避免在边界推断过程中暴露对抗性内容，并能在无人工标注的情况下部署 UCM。

### 2.3 算法流程（文字描述）
1. **页面预处理**：根据显式标签或自动推断的 CSS 选择器，识别不可信 DOM 元素。
2. **掩蔽渲染**：将不可信元素的内容遮蔽，生成占位符并注入 `qllm_id`。
3. **代理交互**：代理观察带占位符的页面并执行操作。若任务需要读取不可信内容：
   - 代理调用 Q-Model，指定目标 ID、问题和返回类型。
   - 系统提取对应元素的原始内容，发送给 Q-Model 实例。
   - Q-Model 返回符合类型约束的答案（如 `{"result": true, "type": "bool"}`），代理接收后继续任务。
4. **敏感任务回退**：若代理无法用类型约束查询完成任务（如需要自由文本），可声明任务不可解，转而请求允许字符串输出的 Q-Model，但输出须经用户批准后方可进入代理上下文。

## 6. 主要结论与发现

- **安全性**：UCM 在增强 WASP 攻击下实现**0% 攻击成功率**，而基线（无防御或提示防御）仍有 8-17% ASR。安全保证由设计提供，不依赖攻击强度。
- **效用保持**：对于不需要不可信内容的任务，UCM 完全保持任务完成率；对于需要不可信内容的任务，代理依赖 Q-Model 同样能保持高效用，在 GitLab 上字符串回退机制可恢复全部效用。
- **成本开销**：中位成本增加 **1.05× ~ 1.84×**，且随着模型能力提升，绝对成本有望下降。在约 15% 的任务中，UCM 甚至由于视觉干扰减少而**降低了成本**。

- **数据流攻击的局限性**：控制流攻击完全阻断，但数据流（如误导代理选择错误项目）依然可能，需结合下游安全策略进一步缓解。

## 8. 不足与局限

- **信任假设过强**：假设网站所有者诚实且正确标注了不可信区域（或推断完全准确）。现实中误标或恶意网站可能绕过防御（但论文指出访问恶意网站本就不安全）。
- **数据流攻击未彻底解决**：攻击者仍可能通过操纵 Q-Model 输出返回错误但类型正确的值（如篡改布尔判断），从而误导代理逻辑。论文承认此局限，并建议结合动作级安全策略。
- **字符串输出需要人工介入**：当任务必须提取自由文本时，需要用户审批 Q-Model 输出，破坏了自动化流程，也增加了用户的负担和风险（用户可能不慎批准恶意内容）。

- **成本与延迟**：虽然开销适中，但对于高频、实时任务，API 成本和多一次 Q-Model 调用的延迟可能仍构成负担。
- **实验覆盖度**：
  - 未在动态、重 JavaScript 的单页应用（SPA）上测试掩蔽机制的鲁棒性。
  - 仅评估了单个网站的 GitLab，其他 WebArena 站点（如电商、论坛）未测试。
  - 未与其他沙箱化方案（如 ceLLMate）进行直接比较。
  - 重复次数较少，部分异常任务可能导致统计波动。
