## 摘要
大语言模型促成了强大的代码补全系统，通过预测后续代码行来辅助开发人员。然而，这些模型仍易受后门攻击，恶意的微调数据会暗中植入不安全行为。尽管防御技术不断进步，但自适应且复杂的后门攻击仍能逃避检测和缓解。我们提出了 CodeTracer，一个将恶意代码补全追责到其背后后门微调数据的取证框架。在现实的部署后约束下运行，CodeTracer 完全依赖微调语料库和报告的误补全事件。它从受损输出中提取结构化的行为指纹，将搜索范围缩小到语义相关的代码样本，并利用基于大语言模型的推理将不安全逻辑归因到特定的后门数据。在三个代表性漏洞案例和十种后门攻击，以及十六个竞争基线上进行的广泛评估表明，CodeTracer 持续实现了高取证准确率、低误识别率，并对自适应攻击表现出强鲁棒性。

## Abstract
Large language models have enabled powerful code completion systems that assist developers by predicting subsequent lines of code. However, these models remain vulnerable to backdoor attacks, where malicious fine-tuning data covertly implants unsafe behaviors. Despite advances in defensive techniques, adaptive and sophisticated backdoor attacks still evade detection and mitigation. We present CodeTracer, a forensic framework that traces malicious code completions back to the backdoor fine-tuning data responsible for them. Operating under realistic post-deployment constraints, CodeTracer relies solely on the fine-tuning corpus and the reported miscompletion event. It extracts a structured behavioral fingerprint from the compromised output, narrows the search to semantically relevant code samples, and employs LLM-based reasoning to attribute unsafe logic to specific backdoor data. Extensive evaluations across three representative vulnerability cases and ten backdoor attacks, along with sixteen competitive baselines, demonstrate that CodeTracer consistently achieves high forensic accuracy, low false identification rates, and strong robustness against adaptive attacks.

### 1. 论文的核心问题与整体含义
*   **研究动机与背景**：大语言模型（LLM）驱动的代码补全系统（如GitHub Copilot）在提升开发效率的同时，也面临着后门攻击的严峻威胁。攻击者通过在微调数据中注入包含触发器和恶意载荷的样本，可使模型在特定上下文下生成带有安全漏洞的代码。尽管已有静态分析、异常检测等防御手段，但面对日趋复杂、隐蔽的自适应攻击（如动态导入、Base64编码混淆），它们常常失效。
*   **核心问题**：传统的防御思路侧重于“预防”或“检测”攻击，但难以根除。本文提出了一个全新的视角——**事后取证归因（Post-hoc Forensic Attribution）**，即在恶意代码补全事件发生后，追溯究竟是微调数据集中的哪些特定样本“教会”了模型这一恶意行为。这有助于诊断模型问题、审计数据管道，并在无需完全重训练的情况下剔除有害数据。

### 2. 论文提出的方法论：CodeTracer
CodeTracer 是一个无需梯度、无需攻击者先验知识的取证框架，仅依赖微调语料库和已发生的误补全事件。其核心思想是：恶意补全必然继承了下毒样本的语义与结构特征，可将这些特征固化为“行为指纹”，并以此在海量微调数据中精准定位源头。
该方法分为三个阶段：
*   **第一阶段：指纹提取 (Fingerprint Extraction)**
    *   利用一个外部LLM（如GPT-4.1）分析误补全事件（包括代码提示和恶意补全），提炼出一个结构化的JSON指纹。
    *   指纹字段包括：漏洞类型（`issue type`）、规范化利用模式（`canonical form`）、核心安全语义描述（`normalized semantics`）、等价语法变体（`equivalent variants`）、攻击者常用的混淆变换（`attack transformations`）以及具体的恶意代码片段（`example extractions`）。
    *   该指纹屏蔽了变量名、格式等表面噪声，是恶意行为的一种紧凑、抗干扰的语义签名。
*   **第二阶段：取证范围缩小 (Forensic Scope Narrowing)**
    *   从指纹中提取原始恶意代码片段 `ĉ`，通过预训练的代码编码器（如 UniXcoder）将其映射为嵌入向量。
    *   将微调数据集中每个样本 `d` 切分为函数级（或固定长度）的代码片段 `d_i`，计算样本内所有片段与 `ĉ` 的最大余弦相似度，得到该样本的相关性评分 `H(ĉ, d)`。
    *   按评分检索并保留 **Top-K** 个候选样本，构成后续分析的“取证范围”，从而避免对全量数据进行昂贵的LLM推理。默认 K=500。
*   **第三阶段：取证归因分析 (Forensic Attribution Analysis)**
    *   对取证范围内的每个候选样本 `s`，使用另一个精心设计提示（Prompt 2）的LLM，要求其将样本与指纹 `q` 进行语义级比对。
    *   LLM 需判断 `s` 是否表达了与 `q` 相同的不安全逻辑。若输出 `[Label: Yes]`，则该样本被认定为后门源头；否则视为干净样本。

### 8. 不足与局限
*   **对外部LLM的依赖**：框架的有效性强依赖作为“法官”的外部LLM的能力。若LLM本身存在偏见或被对抗性输入欺骗（尽管已证明鲁棒性较强），可能会导致归因失败。
*   **潜在的攻击面：虚假报告**：和所有取证系统一样，若攻击者恶意提交大量虚假的“误补全”报告，可能会淹没或误导取证流程。论文建议通过人工审核缓解，但引入了人力开销。
*   **实验范围的可扩展性**：实验主要聚焦于Python语言和三种特定类型的Web/网络安全漏洞。对于强类型语言（如Java, C++）或者算法逻辑型的后门（而非简单的API误用），指纹提取和语义匹配的能力尚待验证。
*   **多文件依赖问题**：虽然测试了多跳攻击，但真实世界中复杂的后门可能分散在多个文件或模块中，当前方法主要针对单个微调样本的归因。
