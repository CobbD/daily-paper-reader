## 摘要
基于大型语言模型的个人AI智能体能够利用可用工具进行推理和行动，以访问电子邮件、管理日历、推送代码至远程仓库，且几乎无需人工监督。当配备长期记忆后，智能体能够回忆起与当前任务相关的具体细节，从而减少对大上下文窗口的需求。目前，长期记忆智能体通常分为两个不同领域：对话型智能体和行动规划型智能体。个人助手智能体处于这两个领域的交汇点，处理敏感信息的同时与不可信的信息源交互，这带来了以往未被考虑的安全漏洞。在本研究中，我们引入了一种新型攻击向量 GhostWriter，它利用工具型个人智能体中的现有记忆子系统来毒化其记忆存储。GhostWriter 分两个阶段运行：注入阶段，攻击者向目标智能体发送隐蔽攻击负载；激活阶段，被毒化的记忆被检索。我们展示 GhostWriter 对最先进的智能体实现了约 98% 的近普遍注入率和约 60% 的高平均激活率。这种攻击之所以可能，是因为缺乏以安全为重点的记忆治理。为此，我们提出了智能体记忆哨兵（AM-Sentry），它利用两种缓解技术：记忆保存策略和记忆检索筛查。实验表明，AM-Sentry 大幅降低了 GhostWriter 的成功率，同时保持了智能体的效用。

## Abstract
Personal AI agents powered by large language models can reason and act using available tools to access emails, manage calendars, and push code to remote repositories, all with minimal oversight. When augmented with long-term memory, an agent can recall specific details relevant to the current task, reducing the need for large context windows. Currently, long-term memory agents tend to fall into two distinct domains: conversational and action-planning agents. Personal assistant agents sit at the convergence of these two domains and handle sensitive information while interacting with untrusted information sources, creating previously unaccounted security vulnerabilities. In this work, we introduce the novel attack vector, GhostWriter, which exploits current memory subsystems in tool-using personal agents to poison their memory store. GhostWriter operates in two phases: injection, where an adversary sends a hidden attack payload to the target agent; and activation, in which the poisoned memory is retrieved. We show that GhostWriter achieves near-universal injection rates of approximately 98% and a high average activation rate of approximately 60% against state-of-the-art agents. This attack is possible due to the lack of security-focused memory governance. In response, we propose Agentic Memory Sentry (AM-Sentry), which leverages two mitigation techniques: a memory-saving policy and a memory-retrieval screen. Our experiments show that AM-Sentry dramatically reduces GhostWriter's success rate while preserving agent utility.

- **背景**：LLM 驱动的个人 AI 代理能够自主处理邮件、日历、代码提交等任务。引入长期记忆后，代理可以跨会话回忆信息，提升任务连续性。但当前记忆系统主要关注效用，缺乏安全治理。
- **问题**：个人助理代理同时处于对话与行动规划领域，处理敏感信息且接触不可信输入源（如外部邮件、文档），这引入了间接记忆中毒的安全隐患。攻击者可通过伪装成正常通信的不可信输入，将恶意负载植入代理记忆，并在后续良性查询时诱导恶意行为（如泄露信息、篡改收件人）。
- **现有工作不足**：已有攻击（AgentPoison、MINJA）大多要求攻击者直接访问记忆或代理；Zombie Agents 等间接攻击假设过强且未在主流代理上验证。现有防御也多针对直接攻击，缺乏对间接、跨会话记忆中毒的防护。

### 2. 提出的方法论

#### 2.1 GhostWriter 攻击向量

- **两阶段攻击协议**：
  - **阶段一：记忆注入（Memory Injection）**  
    攻击者选定目标（完整性破坏/敏感信息泄露/隐蔽泄露/范围升级），构造恶意负载并嵌入看似正常的邮件或日历邀请中，发送至目标用户。代理通过工具接口摄入后，负载被保存至长期记忆。注入成功判定标准为负载在记忆中被完整保留。
  - **阶段二：攻击激活（Attack Activation）**  
    用户提交良性提示（如“发邮件给Dmitri”），代理从记忆中检索相关条目。若恶意记忆被检索并进入上下文，代理行为即被导向攻击者意图。
- **黑盒优化**：为提高检索命中率，攻击者可利用公开邮件语料（如Enron）聚类话题，将负载重写为与目标提示高度相似的文本，并迭代优化直至相似度超过阈值（见公式1）。

#### 2.2 AM‑Sentry 防御框架

- **设计目标**：保持效用、签名无关、恶意记忆表征、信任源建模、架构无关、两阶段覆盖（保存时与检索时）。
- **记忆保存策略（三种严格度）**：
  - **S1**：仅由一个 LLM 法官判断记忆是否对未来任务有用，是/否二分决策。
  - **S2**：LLM 对记忆的四个维度（来源、信任度、类型、效用）评分（0‑1），再由非 LLM 逻辑计算有用性分数 U 和脆弱性分数 V，阈值控制准入。
  - **S3**：最严格，使用七项检查清单（含关键项：脆弱性、敏感性、来源信任；非关键项：矛盾、指令、可验证性、理由），先硬拒绝脆弱性高分项，再综合评分，最后引入第二个 LLM 法官检测首轮是否被注入攻击影响。
- **检索筛查（Retrieval Screen）**：
  - 在记忆检索后、注入上下文前，LLM 法官评估每条记忆的相关性、指令抑制、信任过滤、矛盾检测。若记忆包含来自不可信源的非可验证声明或行为指令，则被滤除。
  - 可与 S2/S3 的结构化元数据（如 `[trust=external-untrusted, risk=medium, type=instruction]`）结合，强化判断。

### 6. 主要结论与发现

- GhostWriter 注入成功率平均约 98%，激活成功率平均约 60%，对多数代理高度有效；轨迹型记忆（ExpeL）相对更鲁棒，但也仍达 28‑63%。
- 代理因缺乏安全记忆治理，轻易保存并信任外部输入，使得间接中毒极易成功。
- 提示注入攻击对这些长期记忆代理的激活率整体较低（多数接近 0%），表明 GhostWriter 具有不同的攻击特征和更大的实际威胁。
- AM‑Sentry 显著降低攻击成功：S3 策略将平均注入成功率降至约 15%，结合检索屏后端到端成功率在多数模型低于 12%。
- 效用损失极小：F1 分数、LLM 评判分数与基准相比仅轻微下降，工具准确率甚至在一些严格配置下略有提升，表明安全与实用可平衡。

### 7. 优点

- **新颖的攻击建模**：首次系统性地研究通过不可信工具输入间接毒化长期记忆的威胁，提出灰盒/黑盒下的优化方法。
- **与现有攻击的对比**：明确区分 GhostWriter 与提示注入的攻击特征，展示现有提示注入检测难以拦截描述型记忆中毒。
- **实用防御设计**：AM‑Sentry 模块化、无签名依赖，兼顾记忆准入和检索阶段，提供渐进严格的配置以适应不同安全需求。
- **全面的评估**：同时考量攻击有效性、防御效果与效用影响，覆盖 5 种代理和 4 种 LLM，实验丰富。
- **提供的工具包**：计划公开代码、攻击场景和效用测试套件，利于后续研究。

### 8. 不足与局限

- **攻击面受限**：仅测试邮件与日历邀请通道，未涉及文档、网页、代码库等其他输入源。

- **代理覆盖有限**：虽选了 5 种主流代理，但记忆架构空间更广，某些设计的漏洞模式可能不同。

- **对抗性假设**：未考虑自适应攻击者可能针对防御清单和检索屏进行规避。
